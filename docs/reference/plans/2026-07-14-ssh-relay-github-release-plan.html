Draft SSH relay Regression gated

Ship the relay as a verified runtime, not a remote npm project.

Publish self-contained, platform-specific relay runtimes through the same GitHub Release used by the Orca desktop app. Orca downloads and verifies the matching runtime locally, transfers it over the existing SSH connection, and launches it from an immutable remote cache.

✓

Recommended decision

Make the verified runtime the primary path, keep the existing embedded relay plus remote Node/npm installer as an automatic fallback only for classified availability and compatibility failures, and remove or narrow that fallback only after the complete declared support matrix has passed and soaked. Build and sign relay artifacts before desktop packaging so every app embeds the exact signed tuple-to-content manifest needed for offline remote-cache reuse.

Target architecture

The SSH host never needs GitHub access.

The desktop owns release access, signature verification, caching, and extraction. SSH authentication/connection and relay RPC transport stay unchanged. Bootstrap artifact transfer is materially redesigned, independently gated, and still travels only through the authenticated SSH connection.

  1. 01 GitHub Release

    Relay build precedes desktop builds and emits the signed manifest plus exact archives.

  2. 02 Local cache

    Embedded manifest resolves identity offline; downloaded bytes are verified and extracted atomically.

  3. 03 SSH transfer

    Bounded streaming over SFTP or system SSH; remote tar is an optional fast path.

  4. 04 Remote cache

    Match the embedded content ID, verify transferred files once, record proof, then launch.

What “100% validated” means

Every required cell in the declared two-layer, risk-based matrix has evidence. It does not mean every possible client × SSH server combination. A tuple without real-environment proof stays on the legacy path and is not enabled by default.

Baseline

Change the dependency boundary without changing relay behavior.

The protocol, SSH authentication/connection, relay RPC transport, immutable install directories, locks, reconnect logic, and garbage collection stay in place. Packaging and bootstrap artifact transfer are the moving parts.

Today

  • Orca embeds platform-neutral relay JavaScript.
  • The remote supplies Node and npm.
  • Remote npm installs node-pty and @parcel/watcher.
  • Linux may require Python, make, a compiler, and registry access.
  • Incomplete Node/npm pairs reproduce issue #8450.

Target

  • GitHub stores a complete, signed relay runtime per tuple.
  • The desktop downloads, verifies, and caches it.
  • POSIX targets require a declared POSIX sh/byte-stream/file-operation baseline; Windows targets require a declared PowerShell/.NET baseline.
  • The authenticated account needs permission to write and execute; Node, Python, npm, and tar are not prerequisites.
  • Bundled Node loads native modules built against that exact runtime.
  • The existing installer remains available if compatibility probing fails.
Implementation

Eight reviewable phases, no big-bang switch.

Each phase has an independently testable output. Early phases add evidence and artifacts without changing the default SSH connection path.

Stabilize the legacy fallback

Finish the narrow #8450 change so Node and npm are selected as one coherent toolchain. Keep it separate from runtime distribution so affected users receive a small, reviewable fix first.

1

Define the runtime contract

Specify archive layout, manifest schema, platform/libc selection, Node version, protocol identity, dependency versions, licenses, file modes, support baselines, and exact POSIX/Windows bootstrap primitives. Hash the full runtime contents rather than only relay JavaScript.

OutputVersioned schema plus pure selector and identity tests.
2

Build self-contained runtimes

Assemble pinned Node, relay entries, patched node-pty, the exact @parcel/watcher package, and required JavaScript dependencies on matching CI environments. Hosted Linux jobs first prove the preinstalled compiler/toolchain with executable C and C++ probes; only genuinely missing prerequisites use HTTPS with bounded connection timeouts and retries. Probe or install exhaustion fails the native build before any artifact is accepted. Run executable runtime smoke probes before archiving. POSIX relay outputs use deterministic .tar.br archives produced and consumed with Node's built-in Brotli streams at quality 9, lgwin=20, and 64 KiB chunks; Windows relay outputs remain ZIP. This lets every desktop client extract every remote tuple without a system XZ tool, a packaged native decompressor, or the 64 MiB XZ dictionary that exhausted the desktop extraction budget.

OutputLocally verifiable archives for every enabled tuple.
3

Sign and publish release assets

Use target-native relay build jobs plus native signing jobs. A fail-closed aggregate job waits for every signed macOS/Windows byte, computes final hashes, signs the canonical manifest, and makes its immutable output a prerequisite of every desktop packaging job. Upload those exact archives, manifest, and signature to the existing draft GitHub Release. Extend the required-assets gate to verify presence, size, signature, manifest coverage, byte identity, and read-back before publication; timeout, signing failure, exhausted retry, or manual-approval failure leaves a recoverable draft and blocks every consumer.

OutputAtomic app-and-relay release publication.
4

Add the desktop artifact resolver

Resolve the remote tuple and content ID from the embedded signed manifest, use its exact build tag and direct release URL, download through Electron networking, verify before extraction, and atomically populate a content-addressed cache. Coordinate cache ownership across app processes and preserve ORCA_RELAY_PATH for development.

OutputSafe cold, warm, concurrent, and offline cache paths.
5

Deploy and validate the runtime

Replace whole-tree buffering and serial-only assumptions with bounded file streaming: capability-probed tar may accelerate POSIX system SSH, but no-tar file streaming is required and uses the SSH channel as a byte-preserving stdin stream into exclusive per-file staging with only the declared shell/file primitives; Windows uses a bounded binary PowerShell/.NET path and must not materialize the runtime as one JSON/base64 buffer; SFTP uses bounded concurrency. After locally verifying signed metadata and archive/tree hashes, transfer through authenticated SSH, restore every declared executable bit, then execute the transferred bundled Node to hash the entire staging tree. Verify the full tree, run bundled Node/native-module and minimal PTY/watcher probes, write a structured .install-complete containing schema, mode, and digest, atomically publish the immutable install, and only then launch. Preserve current locks, reconnect, and garbage collection.

OutputPrimary path performs no remote npm or HTTP operation.
6

Make fallback explicit and observable

Add auto, bundled, and legacy modes. The public Beta is a per-SSH-target bundled-auto choice that maps to auto; absent or unknown configuration maps to legacy, and forced bundled remains an engineering/support diagnostic. Record privacy-safe stage and fallback reason codes in diagnostics, without hostnames, usernames, or paths. Give every stage a budget and shared AbortSignal. Before fallback, cancel and await all download, extraction, upload, and probe work. Bundled and legacy installs use distinct identities, locks, and sentinels so late work cannot mutate the other mode. Only availability/compatibility failures may enter proven legacy behavior in auto; signature, hash, archive-safety, or native-signature failures fail closed without automatic mode switching.

Publish assets before consuming them, then ship the per-target Beta through RC and stable while every target remains on legacy unless explicitly opted in. Any default-on RC or stable promotion requires a separate reviewed decision. Enable only tuples with complete live evidence; keep the fallback through at least two stable cycles.

Legacy families are not bundled evidence

Current source recognizes Linux, macOS, and Windows remotes on x64 and arm64 and uses coherent remote Node/npm plus the remote native-install path. Promotion rule

Milestone 1 starts with no declared bundled tuples and decides which candidates become enabled. Once enabled, a tuple blocks release if its asset or live proof is missing. Candidates awaiting a trustworthy Node runtime source or real hardware remain supported only if their legacy E2E still proves current behavior. Otherwise they stay ineligible for rollout rather than being silently claimed as supported.

Initial immutable runtime baseline

The first contract pins unchanged official Node v24.18.0 LTS binaries. Linux glibc candidates require kernel 4.18, glibc 2.28, and libstdc++ 6.0.25 with GLIBCXX_3.4.25; macOS candidates require 13.5. Initial Windows x64 candidates require Windows 10 22H2 build 19045 or Server 2022 build 20348, and arm64 requires Windows 11 24H2 build 26100. The immutable manifest's single monotonic minimumBuild is therefore 19045 for x64 (which also admits Server 2022 build 20348 and newer builds) and 26100 for arm64. Windows bootstrap additionally requires OpenSSH for Windows 8.1p1, Windows PowerShell 5.1, and .NET Framework 4.8.

Node's detached checksum signature is verified against a build-input-pinned official release keyring before extraction. Musl has no accepted official binary for this package, so musl remains legacy-only until Orca owns and proves a target-native source build. Rosetta-translated shells remain legacy until the x64 artifact passes a live Rosetta SSH cell. These are selector eligibility constraints, not tuple enablement; every candidate still needs the complete two-layer evidence.

Implementation evidence on 2026-07-14 rejected Linux native modules built directly on Ubuntu 24.04: the resulting pty.node required GLIBC_2.32 and GLIBC_2.34 and could not load in the declared glibc 2.28 userland. Linux native modules must therefore be compiled and smoked in the digest-pinned oldest glibc/libstdc++ userland on the tuple-native architecture. A later baseline-container test cannot repair bytes built against a newer ABI, and that userland proof still leaves the exact kernel 4.18 cell open.

The prepared Rocky 8 builder pins Python 3.9 because node-gyp 12 cannot run under the distribution's default Python 3.6. GCC 8 uses the draft spelling gnu++2a for the C++20 mode requested by Node v24's build headers, so the Linux-only build step must replace exactly one -std=gnu++20 occurrence in the already verified, extracted common.gypi and fail closed on any other count. This build-header compatibility step never changes bundled Node bytes, does not apply to macOS or Windows, and avoids introducing a newer libstdc++ dependency; complete native-module smoke and ABI inspection remain mandatory proof.

The offline Linux build container runs as the native GitHub runner's numeric user and group so exclusive staging written through bind mounts remains runner-owned on both hosted architectures. Its writable /tmp is an explicitly mode-1777 tmpfs; the root filesystem remains read-only and the existing no-network, capability-drop, no-new-privileges, process, memory, and CPU bounds remain mandatory. This avoids both root-owned host output and broad permission changes to runner temporary directories.

Windows target-native native-module builds use three immutable inputs authenticated by that same signed checksum document: the tuple's official Node ZIP, the exact-version headers.tar.gz, and the tuple's win-*/node.lib. The Windows ZIP does not contain build headers or the import library. Stage only the required executable, license, headers, and import library into the local build root; disable implicit node-gyp downloads so no unrecorded build input can enter.

Archive contents
  • Unmodified official Node v24.18.0 executable pinned by its signed binary-archive hash. Windows native builds additionally authenticate the signed headers archive and architecture-specific node.lib, neither of which is shipped in the Windows runtime ZIP.
  • relay.js and crash-isolated relay-watcher.js.
  • Orca-patched node-pty@1.1.0 built for the bundled runtime.
  • Windows also carries the pinned node-pty ConPTY runtime closure (conpty.dll and OpenConsole.exe) for the tuple architecture. Hash both files into the tree identity, classify them as native runtime inputs for signing/trust, and run PTY smoke with the production useConptyDll: true behavior rather than silently exercising system ConPTY.
  • Manifest role cardinality must match the exact target-native closure: Linux has one node-pty native file and no native-runtime files, macOS has one node-pty native file plus spawn-helper, and Windows has both conpty.node and conpty_console_list.node plus conpty.dll and OpenConsole.exe. Release and desktop validators reject either missing or extra role members with the same tuple-specific rules.
  • @parcel/watcher@2.5.6 with exactly one native optional package.
  • Runtime metadata, dependency licenses, and a file-level SHA-256 manifest.
  • No development dependencies, package manager, compiler, source maps, PDBs, or Orca-built debug symbols unless explicitly needed for diagnostics. Preserve the verified official Node executable byte-for-byte even when upstream metadata is present; never strip or rewrite those authenticated bytes.
Release trust and identity
  • The packaged app embeds the signed manifest, exact build tag, and accepted public keys.
  • Relay artifacts are immutable inputs to every desktop build, and those exact bytes are uploaded.
  • Ed25519 signs a fixed-order, schema-validated UTF-8 JSON projection through the existing tweetnacl dependency. Tuple and file arrays are sorted, identity fields are portable ASCII, numbers are safe integers, and signatures remain outside the signed projection.
  • An offline-generated 32-byte seed exists only in a tag-restricted relay-runtime-manifest-signing GitHub Environment with required reviewers. Only the aggregate signing job receives it; runtime publication stays blocked until that environment and its access audit exist.
  • Key IDs are SHA-256 of the raw 32-byte public key. Rotation dual-signs two desktop releases and at least 30 days; revocation and emergency replacement use a new desktop build. Unknown, duplicate, or malformed signature entries fail closed.
  • Every Node/CVE/runtime/key change ships as a new desktop tag/build with a newly embedded manifest; old clients stay frozen to their embedded manifest and accepted keys.
  • There is no mutable freshness channel or runtime-only latest lookup; emergency refresh and key response use the normal desktop update path.
  • The embedded manifest tag/channel/version must equal the compiled desktop identity. A newer desktop rejects an older manifest; an explicit desktop downgrade uses only its own embedded manifest and isolated cache identity.
  • GitHub keyless artifact attestations supplement build provenance but never replace the offline-verifiable embedded Ed25519 signature.
  • Archive identity includes every executable and native artifact.
  • Downloads use /releases/download/<exact-tag>/<asset>, never a mutable latest URL.
  • Signature or hash failure rejects and deletes the artifact before any remote transfer.
  • Preserve valid official Node signatures. Before final hashes, sign every Orca-built macOS executable, .node, dylib, and spawn-helper with the existing Developer ID identity, and send every unsigned Windows .exe, DLL, and .node through the existing SignPath inner-binary flow.
  • Native build/signing jobs verify platform-native trust on native runners and put the attestation plus exact byte hashes in the signed manifest; clients verify that portable signed evidence rather than assuming local tooling for a different target OS.
  • The target repeats native-policy execution probes before finalization; native-trust failure at either boundary fails closed.
Platform and compatibility selection
  • Detect Linux libc without Node: prefer marked getconf GNU_LIBC_VERSION, then parse marked ldd --version output and musl loader paths.
  • Treat conflicting, noisy, absent, or unparseable libc evidence as unknown; auto deterministically selects legacy.
  • Manifest entries declare minimum glibc or musl, libstdc++, kernel, macOS, and Windows build requirements.
  • Probe requirements before download and again by executing bundled Node before install finalization.
  • Cover BusyBox, Alpine, old glibc, containers, Rosetta, startup noise, and ambiguous output in selector and live tests. No BusyBox-only variant is initially qualified: Ubuntu's BusyBox 1.36.1 lacks nohup and selects legacy.
  • The POSIX baseline is POSIX sh with builtin command/printf, byte-preserving stdin-to-file via cat, and required mkdir/rm/mv/chmod/test/nohup semantics. It excludes Node, Python, Perl, tar, base64, checksum tools, and compilers.
  • Windows requires OpenSSH 8.1p1, Windows PowerShell 5.1, and .NET Framework 4.8. A bounded 64 KiB FileStream loop writes raw stdin under an exclusive FileMode.CreateNew staging lock; it rejects early EOF and excess bytes and uses same-volume publish.
  • Both families run a nonce-bound binary-fidelity semantic probe once per authenticated connection. Missing or incorrect primitives are compatibility failures; probe results never survive reconnect or cross host boundaries.
  • The no-tar path opens one bounded stdin byte stream per manifest file and sends exactly the declared size into exclusive staging. It becomes sequential under MaxSessions=1; transferred bundled Node later hashes the complete tree.
Hard resource and latency budgets

The local verified cache is capped at 2 GiB. Archives are rejected above 100 MiB compressed, 350 MiB expanded, 5,000 entries, or 250 MiB for one file. Incremental transfer/extraction memory is capped at 80 MiB on the desktop and 32 MiB remotely; buffers stay at or below 1 MiB. SFTP starts at four files/channels and never exceeds eight open files, dropping to one for MaxSessions=1.

Cold bootstrap is capped at 30 minutes with named sub-budgets, and cancellation must join within 10 seconds. Warm p95 may be at most 100 ms or 10% slower than paired legacy, whichever allowance is larger. Cold p95 may not exceed the smaller of 30 minutes or 115% of measured bytes/throughput time plus five minutes. Offline failures classify within 20 seconds and eligible legacy starts within 10 seconds after classification and join. Native GitHub jobs compare at least ten paired samples under the same network shaping; inconclusive hosted variance requires a dedicated runner rather than a weaker threshold.

Fallback state machine
  1. Resolve the tuple and expected content identity from the embedded manifest.
  2. Run bundled stages under one parent AbortController with separate download, extraction, transfer, and probe budgets.
  3. On an eligible failure, abort the stage, close child processes/channels, and await every task and cleanup promise.
  4. Confirm no bundled lock or partial writer remains, then start legacy deployment under a mode-qualified identity and lock.
  5. Ignore late callbacks by generation; they may report diagnostics but cannot write sentinels, launch, or mutate either install.

Integrity failures, including detected cache mutation, stop before this transition and cannot automatically select legacy. Hostile control of the authenticated remote account is outside the integrity threat model; the remote proof establishes initial transfer/finalization correctness. Warm launches assume Orca's completed directory remains immutable; the sentinel is not ongoing proof against later filesystem mutation. Detected warm-cache mutation fails closed. This does not protect against a remote user who can replace both runtime and sentinel.

Remote verification trust boundary and order
  1. Verify the embedded signature, tuple, limits, archive SHA-256, and extracted tree locally.
  2. Transfer only those verified files through authenticated SSH into an exclusive staging directory.
  3. Do not depend on remote sha256sum or shasum; execute the transferred bundled Node to hash every staged file and compare the complete manifest.
  4. Only after full-tree hashing, native probes, and PTY/watcher smoke may Orca write the structured completion sentinel and publish the install.

Local verification plus authenticated transfer and exclusive staging is the pre-execution integrity boundary. Executing bundled Node is intentionally the first staged binary execution and verifies transfer correctness before completion. A hostile authenticated remote account remains outside scope because it can replace both the runtime and its proof.

Risk-based matrix, not an implied full Cartesian

Evidence has two layers. Every enabled remote tuple must pass both built-in SFTP and system SSH, including Windows remote/system SSH and MaxSessions=1. Every supported client OS/architecture must separately prove resolver/cache plus both transfer families against representative POSIX and Windows remotes. Each required cell has an evidence ID; untested client × remote combinations are not claimed.

GitHub Actions runner policy

GitHub Actions is the primary orchestration and evidence surface for PR, nightly, release, and promotion gates. A GitHub-hosted runner may fill a required cell only when it provides real hardware or native virtualization for the declared OS/architecture and exercises a real OpenSSH daemon, the full-size packaged runtime, and the production transport named by that cell. Same-host and native-container targets qualify only when they preserve the declared remote, SSH, primitive, egress, and isolation conditions; mocks, reduced fixtures, cross-builds, and QEMU-only runs do not qualify as live evidence.

Cross-family client-to-remote cells use workflow-provisioned ephemeral native SSH targets or an approved self-hosted target pool; the remote need not itself be GitHub-hosted. Each job records the runner label and image, native architecture, remote image or snapshot identity, teardown result, commands, metrics, and artifacts. A tuple stays on proven legacy behavior when no qualifying native runner or reachable representative remote is available. Numeric regression comparisons use a repeatable runner class or a dedicated runner when hosted-runner variance would make the agreed threshold inconclusive.

Qualifying jobs pin explicit native labels: ubuntu-24.04 and ubuntu-24.04-arm, macos-15-intel and macos-15, plus windows-2022 and windows-11-arm. GitHub refreshes these images weekly, so each job logs the resolved image version; *-latest and third-party runner labels do not qualify a tuple merely because another repository workflow uses them. Hosted capacity is unreserved, and no approved self-hosted SSH target pool or cross-family endpoint is assumed.

Risk register

Resolve these before default-on.

The largest risks are compatibility boundaries and release coupling, not the GitHub download itself.

Linux baseline mismatch

Build native modules in the oldest supported glibc and libstdc++ userland on the tuple-native architecture; run the complete runtime there, then separately run on the exact oldest kernel before promotion. Reject newer-userland producer output even when it smokes successfully on the producer runner.

Untrusted musl runtime source

Do not invent a binary source. Document provenance and updates, or retain legacy mode for musl until Orca can build it.

Native patch drift

Build from Orca’s lockfile and patch set; assert the patched diagnostic exports and real PTY behavior in every artifact.

Missing release assets

Keep releases draft until all relay assets pass read-back verification; publication depends on the completeness job.

Cold-transfer latency

Cache by content, prefetch last-known target tuples after updates, preserve progress stages, and keep a bounded fallback.

Home directory cannot execute binaries

Probe bundled Node before finalizing. Auto mode falls back to the remote runtime path with a precise diagnostic reason.

Release asset tampering

Verify a signed canonical manifest and archive SHA-256 before extraction; never execute or transfer rejected bytes.

Fallback masks defects

Record stage-specific reason codes and require zero unexplained fallback across the RC promotion window.

Bundled Node lifecycle

Own a supported-version/EOL policy, SBOM and provenance, update cadence, and CVE response SLA. Every refresh ships a new desktop build and embedded manifest; installed old clients remain pinned.

Signing-key lifecycle

Specify canonical bytes, protected-environment access, key IDs, dual-key rotation, revocation response through a desktop update, and exact-tag anti-rollback. Do not add mutable freshness lookup.

OS-native trust controls

Test quarantined bytes on clean macOS 13.5 x64/arm64 snapshots with Gatekeeper and strict codesign. Test Server 2022 x64 and Windows 11 24H2 arm64 with current Defender intelligence, then Windows 11 24H2 x64 with the release WDAC policy in audit and enforced modes. Record snapshot, policy, and security-intelligence IDs; any denial blocks release rather than weakening the endpoint control.

Large-runtime transfer

Use bounded streaming and cancellation per transport; prohibit whole-tree JSON/base64 buffering and require a no-tar path before default-on.

Fallback races

Abort and join every bundled stage before legacy starts; separate mode identities and reject callbacks from stale generations.

Ambiguous host detection

Use marked no-Node probes and conservative version checks. Unknown, conflicting, or unsupported results choose proven legacy behavior.

Rollout

Publish, observe, enable, then simplify.

Artifact availability lands before production consumption. Every later stage remains reversible from the same app build.

Asset-only

Build and publish runtimes, run read-back E2E, leave production on legacy.

Per-target Beta

Stable Beta soak

Future default review

Per-target Beta contract

Store an extensible runtime mode per SSH target rather than a global boolean. Missing, malformed, imported, existing, and newly created target configuration defaults to legacy. The Beta option selects bundled-preferred auto behavior only for that target and only on its next connection or an explicit reconnect; it never hot-swaps a live PTY, watcher, or relay generation.

Classified availability and compatibility failures may enter proven legacy automatically. Integrity and security failures still fail closed, but the local error surface provides an explicit action to disable Beta for that target and reconnect through legacy. Shipping this option does not authorize changing the default or removing legacy behavior.

Definition of done

Stable is blocked until every statement is evidence-backed.

These are release criteria, not aspirations. Missing bundled evidence keeps the affected tuple on legacy only when its legacy E2E passes; otherwise Orca must not claim that tuple as supported.

Functional and compatibility

  • Every enabled tuple passes its real-environment artifact E2E.
  • A host with no Node, npm, compiler, or egress gets working PTYs and file watching.
  • The exact #8450 environment passes bundled and forced-legacy modes.
  • Reconnect, reattach, upgrade, downgrade, and rollback preserve live sessions.
  • All relay RPC families pass conformance through the packaged runtime.
  • Existing relay, PTY, watcher, Git, and release suites remain green.

Release and operations

  • Every asset is signed, hashed, uploaded, read back, and executed before publication.
  • Every desktop embeds the exact signed manifest used to identify offline remote caches.
  • Transfer memory, cancellation, no-tar, permissions, and structured-sentinel gates pass.
  • No primary-path remote command invokes npm or opens an HTTP connection.
  • GitHub failure falls back without corrupting local or remote cache state.
  • Every integrity/security failure fails closed in auto and never starts legacy automatically.
  • Node lifecycle, SBOM, provenance, key rotation, and native signing policies are operational.
  • Each required two-layer matrix cell links an evidence ID; no untested Cartesian combinations are claimed.
  • Warm connections show no meaningful latency regression against the recorded baseline.
  • Three consecutive RCs pass with zero unexplained fallback reasons.
Stop condition

If a required live runner, trusted runtime source, compatibility baseline, or rollback proof is unavailable, do not declare the tuple bundled-by-default. Continue serving it through the tested legacy path.
