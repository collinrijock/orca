# Issue #5919 — Pasting 200+ lines hangs Orca 20+ seconds

**Status:** NOT REPRODUCED (fixed in current tree)

## Summary

The Discord report (2026-06-20) said pasting 200+ lines into a terminal froze
Orca's UI for 20+ seconds. On the current tree this does not reproduce: the
interactive terminal paste path is fully cooperative — it measures the payload
and writes it in bounded chunks while yielding to the event loop, so a large
paste cannot block the renderer thread the way the report described.

## Root cause (of the original hang) and why it no longer applies

The report predates the terminal paste coordinator. That machinery landed in
`terminal-paste-coordinator.ts` on 2026-06-19 (commit "Fix paste ownership, input
bounds, and IPC validation"), one day *before* the report — the reporter was on
an older packaged build whose paste path handed the entire clipboard blob to
xterm synchronously on the main thread.

The current interactive paste path (`TerminalPane.tsx:2104` and `:2125`) routes
every paste through:

- `planTerminalPasteWithYield` (`terminal-paste-coordinator.ts:104`) → measures
  payload metadata via `measurePastePayloadMetadataWithYield`
  (`src/renderer/src/lib/paste-payload-metadata.ts`), which yields to the event
  loop every `PASTE_PAYLOAD_METADATA_YIELD_CODE_UNITS` (256K) code units. The
  scan is O(n) linear.
- `executeTerminalPastePlan` (`terminal-paste-executor.ts:26`) → for payloads
  above `TERMINAL_PASTE_DIRECT_MAX_BYTES` (64 KB) the plan mode is `chunked`
  (`terminal-paste-coordinator.ts:162`), and the executor writes bounded
  16 KB UTF-8 chunks (`terminal-paste-chunks.ts`) with
  `await yieldToEventLoop()` between every write (`terminal-paste-executor.ts:104`),
  plus a 30 s per-operation timeout guard (`terminal-paste-limits.ts:4`).

A "200 line" payload (~10–16 KB) stays under the 64 KB direct threshold, so it is
a single non-blocking write; larger pastes are chunked + yielded. No path re-scans
the payload superlinearly.

## Automated evidence

A unit test imports the real coordinator/executor/metadata modules and pins the
non-blocking contract (cooperative metadata scan; chunked, yield-between-writes
execution for large payloads).

Re-run:

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/components/terminal-pane/repro-5919-large-paste-nonblocking.test.ts
```

Result: 2 passed. The tests assert the FIXED behavior (yields > 1 during metadata
measurement; `plan.mode === 'chunked'`; one `yieldToEventLoop` per PTY write;
the whole-blob `pasteText` fallback is never taken for a chunked plan).

## Manual repro (current build)

Paste 200+ lines into a terminal pane with Cmd/Ctrl+V. The paste completes
without a multi-second UI freeze. (Any remaining latency for very large pastes is
the target program echoing/rendering output, not Orca blocking on the paste.)

## Fix direction

None required — the async paste coordinator already resolves the reported hang.
If a regression is ever suspected, the guardrails to preserve are: the yielding
metadata scan, the 64 KB direct threshold, chunked writes, and the
`await yieldToEventLoop()` between chunks.
