# Issue #7905 — Unable to use the opencode terminal (blank pane)

**Status:** NOT REPRODUCED (needs info)

## Summary

The report is: on macOS, selecting **OpenCode** for a terminal yields a blank
pane while other agents render fine. The issue carries no Orca version, no logs,
and no confirmation that the `opencode` binary is actually installed/on PATH for
the launched PTY.

I traced the full opencode launch/command-construction path in the current tree
and could not find a logic defect that would blank the pane. The launch command
Orca produces for opencode is correct and covered by passing tests. The
"blank pane" symptom is a runtime behavior of the real `opencode` binary inside
the PTY, which is not reproducible from the code alone with the information
given.

## What I verified

1. **Launch command is correct.** `src/shared/tui-agent-config.ts:133` defines
   opencode with `launchCmd: 'opencode'`, `expectedProcess: 'opencode'`, and
   `promptInjectionMode: 'flag-prompt'`. In `src/shared/tui-agent-startup.ts`:
   - No prompt (terminal picker path, `allowEmptyPromptLaunch`): launchCommand is
     the bare `opencode` (line 89).
   - With a prompt: `flag-prompt` produces `opencode --prompt '<text>'`
     (`src/shared/tui-agent-startup.ts:114-124`). This is the intended contract
     from PR #6798 ("Reliably deliver the issue prompt into opencode's composer")
     and is pinned by existing tests.

2. **No unsupported flags are injected.** opencode is intentionally *excluded*
   from `YOLO_TUI_AGENT_ARGS` (`src/shared/tui-agent-permissions.ts:6-31`), so it
   gets no default extra args. `src/shared/tui-agent-launch-defaults.ts:5-8`
   further lists `--dangerously-skip-permissions` as unsupported for opencode and
   strips it if a user configured it. So opencode never launches with a
   Claude-style flag that could make it error out and clear the screen.

3. **No opencode-specific renderer/detection gate hides output.** A repo-wide
   search found no code path in `src/renderer/src/components/terminal-pane/` or
   `src/main/` PTY/runtime code that suppresses or blanks output specifically for
   opencode. opencode-specific logic is limited to title parsing
   (`src/shared/opencode-terminal-title.ts`), completion/interrupt inference, and
   usage/rate-limit scraping — none of which affect whether the pane renders.

4. **Existing launch tests pass on the current tree**, confirming the produced
   command (see re-run command below).

## Root cause

Not identified in code. The most likely real-world causes, none of which are a
findable logic defect in the current tree, are:

- `opencode` is detected by the agent picker (login-shell PATH) but is not on the
  PATH of the launched PTY, so the command fails and the alt-screen TUI leaves a
  near-empty pane. (Speculative — would need PTY logs.)
- A version/behavior mismatch in the installed `opencode` binary vs. the
  `--prompt` flag contract.

`command not found`, the fix is a PATH/detection-parity concern (detect and
launch must resolve the same binary). If logs show opencode erroring on
`--prompt`, revisit the `flag-prompt` contract in `tui-agent-config.ts`.
