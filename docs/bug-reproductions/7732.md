# Issue #7732 — GitLab pipeline job details never load in the Checks side panel

**Status:** REPRODUCED

## Summary

For a GitLab MR, the Checks side panel lists each pipeline job with the correct
status, but expanding a job row never shows its log/details — it renders
"No inline details are available for this check." The mapping that turns GitLab
jobs into check rows drops the GitLab job id, so the expand path has no handle to
fetch the job trace and falls back to the GitHub loader, which returns `null`.

## Root cause

`gitLabPipelineJobsToPRChecks` in `src/shared/gitlab-pipeline-checks.ts:60-67`
maps each `GitLabPipelineJob` to a `PRCheckDetail` carrying only
`name`/`status`/`conclusion`/`url`. The source job has `id: number`
(`src/shared/gitlab-types.ts:279`), but the mapping discards it and never sets
`checkRunId`/`workflowRunId`. `PRCheckDetail` (`src/shared/types.ts:1310-1329`)
has no GitLab job field at all.

When a row is expanded, `requestCheckDetails` in
`src/renderer/src/components/right-sidebar/checks-panel-content.tsx:1041-1059`
routes on `checkRunId`/`workflowRunId` and hands off to the GitHub-only
`getPRCheckDetails` (`src/main/github/client.ts`), which needs owner/repo plus a
check-run/workflow id. A GitLab-mapped row has none, so the detail resolves to
`null` and no trace is ever loaded — even though the GitLab backend
(`gitlab:jobTrace` → `getJobTrace` in `src/main/gitlab/client.ts`) already exists
and is used by `GitLabItemDialog.tsx`.

## Automated evidence

The repro test imports the real product mapping and asserts the mapped row

`workflowRunId`, no numeric handle) despite the input job having `id: 42`. It

## Manual repro

1. Link a GitLab MR whose head pipeline has at least one failed job.
2. Open the Checks side panel — job rows appear with correct statuses.
3. Expand any job row.
4. Observe "No inline details are available for this check."

## Fix direction

1. Add an optional `gitlabJobId?: number` to `PRCheckDetail` and populate it in
   `gitLabPipelineJobsToPRChecks` from `job.id`.
2. In `ChecksPanel`, when `activeGitLabReview` is set, supply a GitLab-specific
   `onLoadCheckDetails` that calls `window.api.gl.jobTrace({ jobId, projectRef, ... })`
   and adapts the trace into a `PRCheckRunDetails` (trace text → rendered log),
   mirroring the existing `GitLabItemDialog` handler. Keep the GitHub path
   unchanged and guard the GitLab branch behind `activeGitLabReview`. Must keep
   working over SSH/remote (trace goes through the existing runtime RPC path).
