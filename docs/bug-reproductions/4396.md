# Issue #4396 — Linear and GitHub issue Use actions force new workspace setup

**Status:** PARTIAL (Linear half reproduces; GitHub half already fixed on current tree)

## Summary

The issue reports that the normal **Use / Start** actions for existing GitHub and
Linear issues route through the New Workspace composer instead of a direct launch
path, forcing the user into worktree setup.

On the current tree (`b8502b21bb`) this is only **partially** true:

- **GitHub — FIXED.** `handleUseWorkItem` now routes through
  `createGitHubWorkItemWorkspaceInBackground`, a direct background-create path
  that does **not** open the composer in the happy case (it only falls back to the
  composer for degraded cases: repo missing, host unavailable, setup policy
  `ask`, PR start-point failure, agent-startup failure). This was reworked after
  the issue was filed (issue pinned commit `9b5bfbafb7`).
- **Linear — STILL BUGGY.** `handleUseLinearItem` routes straight to
  `openModal('new-workspace-composer', ...)`, forcing New Workspace setup for an
  existing issue. There is no attachment reuse and no direct/background path,
  even though `launchWorkItemDirect` already supports Linear issues via
  `linearIdentifier`.

## Root cause

Linear Use action forces the composer (`src/renderer/src/components/TaskPage.tsx`):

- `TaskPage.tsx:8012` `handleUseLinearItem` records telemetry and calls
  `openComposerForLinearItem(issue)` — nothing else.
- `TaskPage.tsx:7999` `openComposerForLinearItem` calls
  `openModal('new-workspace-composer', ...)` (`TaskPage.tsx:8002`).
- The Linear list/detail surfaces wire `onUse` → `handleUseLinearItem`
  (`TaskPage.tsx:10180`, `:10793`, `:11047`), so every Linear Use/Start entry
  point lands on the composer.

The direct path exists but is never wired to the Linear action:

- `src/renderer/src/lib/launch-work-item-direct.ts:50` `launchWorkItemDirect`
  supports Linear via `item.linearIdentifier`
  (`launch-work-item-direct.ts:128`) and `getLinearIssueWorkspaceName`.
- `launchWorkItemDirect` is imported only by `PullRequestPage.tsx`,
  `github-project/ProjectViewWrapper.tsx`, and `lib/fix-checks-agent-launch.ts`.
  `TaskPage.tsx` never imports it, so no TaskPage Use action can reach it.

GitHub contrast (fixed): `TaskPage.tsx:6771` `handleUseWorkItem` calls
`createGitHubWorkItemWorkspaceInBackground({ ... openModalFallback: () =>
openComposerForItem(item) })` (`src/renderer/src/lib/github-work-item-background-create.ts:160`),
whose happy path returns `{ kind: 'background-started' }` without opening the
composer.

The stale `Why` comment at `TaskPage.tsx:8014` claims the Linear behavior mirrors
`handleUseWorkItem` ("open the New Workspace dialog pre-filled rather than
yolo-creating the worktree") — but the GitHub `handleUseWorkItem` it references
no longer opens the composer, confirming the Linear path was left behind.

## Manual repro (live app)

1. Connect Linear and open the Tasks page.
2. On any existing Linear issue, click the Use / Start-workspace action.
3. Observed: the New Workspace composer opens, forcing name/agent/setup
   confirmation before the workspace is created.
4. Compare with an existing GitHub issue's Use action: it stages the workspace
   in the background (sidebar) without opening the composer.

## Fix direction

Wire the Linear Use action to the direct/background path, mirroring GitHub:

- Route `handleUseLinearItem` through `launchWorkItemDirect` (which already
  handles `linearIdentifier`) or a Linear equivalent of
  `createGitHubWorkItemWorkspaceInBackground`, keeping `openComposerForLinearItem`
  only as the degraded-case `openModalFallback`.
- Consider an `handleOpenOrUseLinearItem` variant that reuses an already-attached
  workspace (as GitHub's `handleOpenOrUseGitHubWorkItem` does) before launching.
- Update/remove the stale `Why` comment at `TaskPage.tsx:8014`.
