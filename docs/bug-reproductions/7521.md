# Issue #7521 — Codex session history duplicates because AI Vault scans both system and Orca runtime roots

**Status:** REPRODUCED

## Summary

AI Vault treats the system Codex sessions root (`~/.codex/sessions`) and the
Orca-managed `codex-runtime-home/home/sessions` root as two independent *display*
sources. When the same physical Codex transcript exists under both roots (Orca
bridges/hardlinks system sessions into the runtime home), the scanner emits one
`AiVaultSession` per file path. Because sessions are keyed/deduped by `filePath`
— not by the Codex session UUID or the file inode — the same session shows up
twice in the panel.

## Root cause

- `src/main/ai-vault/session-scanner-source-discovery.ts:61-70` — `discoverAiVaultSessionSources`
  builds the Codex sessions dirs by unioning `options.codexSessionsDir` (system
  `~/.codex/sessions`) with `options.additionalCodexSessionsDirs` (the Orca
  runtime home, supplied by `cached-session-list.ts:52-53`). Both are scanned as
  display sources.
- `src/main/ai-vault/session-scanner.ts:66-83` — every discovered file becomes an
  independent `SessionFileCandidate`; a hardlinked transcript under two roots
  yields two candidates.
- `src/main/ai-vault/session-scanner-accumulator.ts:96` — each session's id is
  `` `${executionHostId}:${accumulator.agent}:${sessionId}:${accumulator.filePath}` ``.
  The `filePath` component makes the two hardlinked copies distinct ids.
- `src/main/ai-vault/session-list-results.ts:59-62` and
  `src/main/ai-vault/session-scanner.ts:134-141` — dedupe is `byId` (Map keyed by
  that path-bearing id), so it never collapses the same session UUID / same inode.

There is no inode-based or sessionId-based dedupe anywhere in the pipeline, so
the duplicate survives to the panel.

## Automated evidence

Unit test imports the real scanner and pins the duplicate (passes on current
tree while asserting the wrong result):

```
pnpm exec vitest run --config config/vitest.config.ts src/main/ai-vault/repro-7521-codex-runtime-home-duplicate.test.ts
```

The test seeds one physical Codex transcript, hardlinks it into both
`codexSessionsDir` and the runtime-home sessions dir (verifying identical
inode), scans, and asserts `codexSessions` has length 2 with two distinct ids
for the same `sessionId`. Result: 1 file / 1 test passed.

## Manual repro (app steps)

1. Open a workspace and run a Codex session through Orca (writes to the managed
   `codex-runtime-home`), or have a session that also lives in `~/.codex/sessions`
   and gets bridged/hardlinked into the runtime home.
2. Open the AI Vault panel.
3. The same Codex session appears twice for the one workspace.

## Fix direction

Treat the Orca Codex runtime home as the single authoritative display source and
use `~/.codex/sessions` only as an import/bridge source (as the issue proposes),
or dedupe Codex candidates by inode (`fs.stat().ino`) / session UUID before
building `AiVaultSession` ids so hardlinked copies collapse to one entry. For SSH
workspaces, scan the remote host's runtime home rather than the client's local
storage.
