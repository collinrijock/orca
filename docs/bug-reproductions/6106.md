# #6106 — SSH terminal loses pre-TUI shell output after Codex tab restore

**Status:** PARTIAL (root cause proven at unit level with the real host serializer; the full live SSH restore path is modeled, not driven end-to-end)

## Summary

On an SSH-backed terminal, shell output printed *before* starting a full-screen
TUI (e.g. `codex`, which enters the alternate screen) disappears from the
terminal's scrollback after the tab is hidden and then restored. This does not
happen for the equivalent **local** terminal — only the remote/SSH path loses
the pre-TUI output.

## Root cause

Local and remote hidden-restore snapshots carry different shapes, and the
restore logic branches on a field the remote path never sets.

`applyMainBufferSnapshot()` in
`src/renderer/src/components/terminal-pane/pty-connection.ts:6520-6539` chooses
its restore write sequence based on `snapshot.alternateScreen` /
`snapshot.scrollbackAnsi`:

- `!snapshot.alternateScreen` (line 6520): writes `\x1b[2J\x1b[3J\x1b[H` and
  then `snapshot.data` — clearing/rewriting the **currently active** buffer.
- `alternateScreen === true && scrollbackAnsi !== undefined` (line 6526): first
  writes `\x1b[?1049l` to return to the **normal** buffer, rebuilds it from
  `scrollbackAnsi`, then re-enters the alt screen. This is what preserves the
  pre-TUI scrollback.

The **local** path (`pty:getMainBufferSnapshot` →
`serializeHiddenOutputRecoveryBuffer` → `serializeHeadlessTerminalBuffer`,
`src/main/runtime/orca-runtime.ts:8014-8041`) returns `alternateScreen: true`
and a separate `scrollbackAnsi`, so it takes the scrollback-preserving branch.

The **remote/SSH** path does not:

1. `serializeBudgetedRequestedSnapshot` folds scrollback *into* `data`:
   `(serialized.scrollbackAnsi ?? '') + serialized.data`
   (`src/main/runtime/rpc/methods/terminal.ts:619`), and the snapshot frame it
   sends (`sendSnapshotFrames`) carries **no** `alternateScreen` / separate
   `scrollbackAnsi`.
2. The remote multiplexer's snapshot resolve type only exposes
   `{ data, cols, rows, seq, source, pendingEscapeTailAnsi }` — no
   `alternateScreen`, no `scrollbackAnsi`
   (`src/renderer/src/runtime/remote-runtime-terminal-multiplexer.ts:106-120`,
   returned via `remote-runtime-pty-transport.ts:865`).

So for a remote pty `snapshot.alternateScreen` is `undefined`, and restore takes
the `!alternateScreen` branch. Because the pane is still showing the codex alt
screen when the restore runs, `\x1b[2J\x1b[3J` clears the **alternate** buffer,
the folded `scrollbackAnsi` is written into the alt buffer (then the trailing
`\x1b[?1049h` is a no-op and the alt frame overwrites it), and the normal buffer
scrollback is never rebuilt — the pre-TUI output is lost.

## Automated evidence

Real host serializer (`HeadlessEmulator.getSnapshot`) + a real renderer xterm
driven through both restore branches. The remote branch loses
`PRE_CODEX_START` / `AGENTS.md`; the local branch keeps them.

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/components/terminal-pane/repro-6106-ssh-alt-scrollback-loss.test.ts
```

Result: 3 passed —
- host snapshot separates pre-TUI scrollback from the alt frame,
- BUG: remote/SSH restore branch discards the pre-TUI scrollback,
- CONTROL: local restore branch (alternateScreen + scrollbackAnsi) keeps it.

## Manual repro (from the issue)

1. Connect an SSH target (`ssh localhost`) and open a remote worktree terminal.
2. In the SSH terminal run: `printf "PRE_CODEX_START\n"; ls | head -5; printf "PRE_CODEX_END\n"`.
3. Start `codex` (enters the alternate screen).
4. Switch to another tab, then back to the codex tab.
5. The restore snapshot shows the codex screen but the pre-codex output/scrollback is gone.

## Fix direction

Carry the alt-screen split across the remote transport so the renderer can take
the scrollback-preserving branch for SSH terminals: have
`serializeBudgetedRequestedSnapshot` / `sendSnapshotFrames` emit
`alternateScreen` and a separate `scrollbackAnsi` (instead of folding scrollback
into `data`), thread them through the remote multiplexer snapshot resolve type
(`remote-runtime-terminal-multiplexer.ts`) and `remote-runtime-pty-transport.ts`
so `applyMainBufferSnapshot` receives `alternateScreen: true` + `scrollbackAnsi`
for remote alt-screen restores, exactly like the local `getMainBufferSnapshot`
path.
