# Issue #1715 — GitHub Projects uses the wrong gh host in multi-host setups

**Status:** REPRODUCED

## Summary

Orca fetches GitHub Issues repo-scoped (cwd = repoPath), so `gh` infers the
host from the checkout's git remote. GitHub Projects calls are made as
hostless, non-cwd `gh api graphql` calls, so `gh` falls back to its globally
active/default host. In a multi-host setup (github.com + a GHES instance) where
the opened repo belongs to the non-default host, Orca checks auth/scopes and
runs project queries against the wrong instance, producing the misleading
"missing project scope" message. Additionally, paste-to-add only accepts
`https://github.com/...` project URLs, so GHES project URLs are rejected.

## Root cause

Three host-agnostic surfaces in `src/main/github`:

1. **Hostless project GraphQL/REST** — `runGraphql` / `runRest`
   (`src/main/github/project-view/internals.ts:265`, `:335`) accept only an
   optional `cwd` and never a `--hostname`. Every project caller
   (`fetchProjectViewsPage`, `fetchAllItems`, `fetchItemsPageWithRaw`
   `src/main/github/project-view.ts:890`, `listAccessibleProjects`,
   `resolveProjectRef`, `resolveOwnerType`, `fetchItemsCountOnly`) invokes them
   with neither cwd nor host, so `gh` uses its global default host.

2. **GHES project URLs rejected** — `parseProjectPaste`
   (`src/main/github/project-view.ts:1576`) hard-codes the URL regex to
   `^https?:\/\/github\.com\/(orgs|users)\/...`, so a GHES project URL never
   matches and paste-to-add returns a validation error.

3. **Host-agnostic auth diagnosis** — `diagnoseGhAuth`
   (`src/main/github/auth-diagnose.ts:93`) takes no host argument and selects
   `accounts.find((a) => a.active) ?? accounts[0]`
   (`auth-diagnose.ts:121`), i.e. gh's global active account, then computes
   `missingScopes` against it (`:127`) — never the repo's host.

## Manual repro (from the issue)

1. `gh auth login` to two hosts: github.com and a GHES instance.
2. Open a repo in Orca that belongs to the non-default (GHES) host.
3. Confirm GitHub Issues load (repo-scoped → correct host).
4. Open GitHub Projects → Orca shows "Your gh token is missing the project
   scope…" even though the GHES token has `project`, because Orca queried and
   diagnosed the wrong host. Pasting a GHES project URL also fails to parse.

## Fix direction

- Thread the repo's host through the project read/write path: resolve the host
  from the repo remote (as Issues already do) and pass it to `gh` via
  `--hostname <host>` (or run with `cwd = repoPath`) in `runGraphql`/`runRest`
  and `fetchItemsPageWithRaw`.
- Generalize `parseProjectPaste` to accept any host, capturing the host so
  downstream calls target it.
- Give `diagnoseGhAuth` a `host` parameter and select the account matching that
  host instead of the global-active one when computing `missingScopes`.

(Open PR #2784 implements multi-host routing along these lines; unmerged.)
