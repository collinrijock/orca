# Issue #7400 — Orca Mobile shows stale one-tab session when main bumps mobile snapshotVersion

**Status:** REPRODUCED

## Summary

A workspace with two live terminal tabs can appear as a single tab on Orca Mobile.
`session.tabs.list` returns only one tab even though the terminal registry, the
renderer graph, and the persisted workspace session all have both terminals. The
missing terminal stays connected and writable.

## Root cause

`RuntimeMobileSessionTabsSnapshot.snapshotVersion` is overloaded for two
different notions of freshness:

1. **Renderer structural freshness** — the renderer maintains its own monotonic
   counter (`src/renderer/src/runtime/sync-runtime-graph.ts`,
   `++mobileSessionSnapshotVersion`) and stamps each structural snapshot (tabs,
   groups, order) with it.
2. **Main-side live freshness** — `touchMobileSessionSnapshotsForPty()` bumps the
   **same** field on every PTY output / title / agent-status notification without
   rebuilding the tab list.

The accept/reject guard in `syncMobileSessionTabs()` compares that single field:

- `src/main/runtime/orca-runtime.ts:21910-21916`
  ```ts
  if (
    !existing ||
    nextSnapshot.publicationEpoch !== existing.publicationEpoch ||
    nextSnapshot.snapshotVersion >= existing.snapshotVersion
  ) {
    this.mobileSessionTabsByWorktree.set(snapshot.worktree, nextSnapshot)
  }
  ```

- The bumping side, `touchMobileSessionSnapshotsForPty()`:
  `src/main/runtime/orca-runtime.ts:4024-4051` (`snapshotVersion: snapshot.snapshotVersion + 1`).

When a stale one-tab snapshot is repeatedly bumped by main-side title/status
churn, its `snapshotVersion` climbs far above what the renderer's structural
counter has reached. A later, correct renderer structural snapshot (same
`publicationEpoch`, containing both terminals, but a lower structural version) is
then rejected by the guard, so the stale one-tab snapshot persists.

## Automated evidence

Unit test that imports the real `OrcaRuntimeService` and drives the real
`touchMobileSessionSnapshotsForPty` + `syncMobileSessionTabs` methods. It passes
on the current tree while asserting the buggy one-tab outcome; a control test
shows the two-tab snapshot IS accepted when main has not bumped the version
(isolating the defect to the shared counter, not the merge logic).

Re-run:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/runtime/repro-7400-mobile-snapshot-version.test.ts
```

Result: `Test Files 1 passed (1) / Tests 2 passed (2)`.

## Manual repro (app-level)

1. Open a worktree with two live terminal tabs, one running a spinner-in-title /
   agent-status agent (e.g. Codex/Claude) that churns its title frequently.
2. Get the runtime into a transient state where the renderer publishes a one-tab
   `mobileSessionTabs` snapshot for that worktree while main keeps bumping it via
   PTY/title/status notifications.
3. Observe Orca Mobile (or a direct `session.tabs.list` RPC) reports one tab and
   an ever-increasing `snapshotVersion`, while `orca terminal list --worktree
   active --json` shows both live terminals. Per the issue comment,
   `session.tabs.createTerminal` also times out in this state.

## Fix direction

Stop using one counter for both meanings. Track renderer structural freshness
separately (per worktree/publication epoch, e.g.
`rendererMobileSnapshotFreshnessByWorktree`) and gate acceptance in
`syncMobileSessionTabs()` on the last accepted **renderer structural** version,
not on `existing.snapshotVersion` after main-side live bumps. Keep a separate
externally-monotonic `snapshotVersion` for mobile/web delivery so live updates
still push, but bumping it must never block a structural tab correction.
