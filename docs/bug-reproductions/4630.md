# Issue #4630 — Claude-Mem Causes Spam of Notifications even if "Main" Agent is Done

**Status:** REPRODUCED

## Summary

With the Claude-Mem plugin installed, Orca fires a second "agent task complete"
notification after the user's turn has already finished. Claude-Mem runs its own
work through Claude's hook stream once the turn is done (a `PostToolUse` event,
which Orca maps to `working`, followed by a `Stop`, which maps to `done`). The
completion coordinator treats that background `working` hook as the start of a
brand-new user turn, so the following `Stop` looks like a freshly completed turn
and re-notifies — even though the user submitted no prompt and the main agent is
idle.

## Root cause

`src/renderer/src/components/terminal-pane/agent-completion-coordinator.ts`,
`observeHookStatus`, the `working` branch at **lines 846-859**:

```ts
if (payload.state === 'working') {
  clearPendingHookDone()
  clearPendingCodexAttention()
  workingStatusObserved = true
  requiresFreshWorking = false
  lastCompletionIdentity = null
  lastAttentionToken = null
  currentTurn += 1          // <-- re-arms the turn for ANY 'working' hook
  dropPendingTitle()
  options.dispatchHookLifecycle?.(payload)
  return
}
```

Any `working` hook bumps `currentTurn` and sets `workingStatusObserved`, with no
distinction between a genuine new user turn (Claude `UserPromptSubmit`, carries a
prompt) and background plugin activity (Claude `PostToolUse`, no prompt). The
per-turn dedup in `dispatchCompletion` (line 302, `lastCompletedTurn === currentTurn`)
is therefore defeated: the background cycle advances `currentTurn`, so the second
`done` passes the guard and dispatches a second notification.

The `PostToolUse -> working` and `Stop -> done` mapping used by Claude-Mem's
events is in `src/shared/agent-hook-listener.ts`, `normalizeClaudeEvent`
**lines 2601-2611**.

## Automated evidence

Coordinator-level unit test that imports the real product module
(`createAgentCompletionCoordinator`) and pins the buggy double-dispatch — it
asserts `dispatchCompletion` is called once after the user turn, then **twice**
after the background Claude-Mem cycle:

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/components/terminal-pane/repro-4630-claude-mem-renotify.test.ts
```

Result on current `main`: `Test Files 1 passed (1) / Tests 1 passed (1)`. The
`toHaveBeenCalledTimes(2)` assertion encodes the wrong behavior; correct behavior
would be `toHaveBeenCalledTimes(1)`.

## Manual repro (app steps)

1. Install Claude-Mem so it runs after each turn via Claude hooks.
2. In an Orca worktree, run a normal Claude turn and let it finish (one
   notification fires — expected).
3. Leave the pane idle. When Claude-Mem posts its background `PostToolUse` then
   `Stop`, Orca fires a second "agent task complete" notification for the same
   pane even though nothing you asked for is running.

## Fix direction

Distinguish a genuine new user turn from background/plugin hook activity before
re-arming `currentTurn`. Options: gate the `working` re-arm on evidence of a real
new turn (e.g. a `UserPromptSubmit`-origin working hook / non-empty prompt) rather
than any `working`, or require a fresh user-prompt boundary before a subsequent
`done` may dispatch another completion for the same pane after one already fired.
Any fix must keep legitimate multi-turn conversations notifying once per real
turn.
