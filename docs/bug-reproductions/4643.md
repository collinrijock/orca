# Issue #4643 — Jira connector: can't create an issue; "Reporter is required", no accountId picker

**Status:** PARTIAL (root cause proven in real code + runnable test; full renderer form path needs the live app)

## Summary

After connecting a Jira Cloud site via API token, creating a **new** issue from Orca
fails with "Reporter is required." When a project marks `reporter` as a required
create field, Orca renders it as a **plain free-text input** (no user-search
picker), and whatever the user types — a display name *or* a pasted `accountId` —
is sent to the Jira REST API as a **bare string** instead of the
`{ "reporter": { "id": "<accountId>" } }` object Jira Cloud requires. Jira rejects
the payload, so the issue can never be created.

## Root cause

Three linked seams, two of them real/imported and one unexported-but-verified:

1. **No user picker in the create form.** `mapCreateField`
   (`src/main/jira/issues.ts:239`) maps a required `reporter` field with
   `schema.type: "user"` and **no `allowedValues`** (user fields never enumerate
   options). The create form's picker branch is gated on
   `field.allowedValues?.length`
   (`src/renderer/src/components/TaskPage.tsx:12565`), so a user field falls
   through to the plain `<Input>` branch (`TaskPage.tsx:12596`). The create form
   never calls `jira.listAssignableUsers` at all — that endpoint is only wired
   into `JiraIssueWorkspace.tsx` for editing existing issues.

2. **Bare-string value for user fields.** `buildJiraCreateFieldValue`
   (`src/renderer/src/components/TaskPage.tsx:1031`) has no `schema.type === "user"`
   branch. Typed/pasted text (name or accountId) falls through to
   `return trimmed` (`TaskPage.tsx:1058`) — a plain string.

3. **createIssue forwards it verbatim.** `createIssue`
   (`src/main/jira/issues.ts:465`) copies custom fields straight into the POST
   body (`issues.ts:485-490`), producing `fields.reporter = "<string>"` instead
   of `{ id: "<accountId>" }`. Jira Cloud rejects a string for a user field.

Contrast `updateIssue` (`src/main/jira/issues.ts:539-550`), which correctly wraps
the assignee user as `{ accountId }` — the create path simply lacks the equivalent
user-field handling.

## Automated evidence

`src/main/jira/repro-4643-reporter-user-field.test.ts` imports the **real**
`listCreateFields` and `createIssue` from `src/main/jira/issues.ts` and pins the
buggy behavior (passes today):

- required `reporter` create field is mapped with `schema.type === "user"` and
  `allowedValues === undefined` → the form renders a plain input, not a picker.
- `createIssue` with the bare-string reporter the UI produces sends
  `fields.reporter === "5b10a2844c20165700ede21g"` (a string), **not**
  `{ id: "5b10a2844c20165700ede21g" }`.

The unexported renderer helper `buildJiraCreateFieldValue` and the JSX picker

form submission is not exercised here — hence PARTIAL rather than a full E2E

## Manual repro (live app)

1. Connect a Jira Cloud site with an API token (import succeeds).
2. Create a new item for a team-managed project that requires a Reporter.
3. The Reporter field is a plain text box with no dropdown/autofill.
4. Type a name or paste a valid `accountId`; submit → "Reporter is required."

## Fix direction

- In the create form, detect `schema.type === "user"` (as it already special-cases
  `option`/`array`/`number`/`textarea`) and render a typeahead backed by
  `jira.listAssignableUsers`, storing the selected `accountId`.
- In `buildJiraCreateFieldValue`, emit `{ id: <accountId> }` for `user` fields.
- Optionally default the reporter to the token owner (the site's `accountId`),
  since the API key already identifies a single user.
