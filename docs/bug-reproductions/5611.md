# Issue #5611 — Copying selected Copilot conversation text shows success but clipboard is unchanged

**Status:** REPRODUCED

## Summary

Selecting text in a terminal/agent pane (e.g. a GitHub Copilot conversation) and
pressing the terminal-copy chord (`Mod+Shift+C`) makes Orca behave as though the
copy succeeded — it consumes the keystroke and, on the right-click path, clears
the selection — but the OS clipboard may never be updated. The copy path fires
`window.api.ui.writeClipboardText(selection)` and:

1. swallows a rejected write with a no-op `.catch`, and
2. never reads the clipboard back to verify the write landed.

Nothing gates success on a verified, non-empty write, so a failed clipboard
write (Windows clipboard busy / access denied — the scenario in the report)
produces a silent no-op that still looks successful.

## Root cause

`src/renderer/src/components/terminal-pane/keyboard-handlers.ts:442-457`
(the `copySelection` branch):

```ts
if (action.type === 'copySelection') {
  const pane = manager.getActivePane() ?? manager.getPanes()[0]
  if (!pane) return
  const selection = pane.terminal.getSelection()
  if (!selection) return
  e.preventDefault()
  e.stopImmediatePropagation()
  void window.api.ui.writeClipboardText(selection).catch(() => {
    /* ignore clipboard write failures */   // <-- failure swallowed, no verify
  })
  return
}
```

The same swallow-and-don't-verify pattern appears on the sibling copy paths:

- `src/renderer/src/components/terminal-pane/use-terminal-pane-context-menu.ts:490-497`
  (right-click copy) — `void writeClipboardText(selection)` then immediately
  `clickedPane.terminal.clearSelection()` without awaiting or verifying.
- `src/renderer/src/components/terminal-pane/use-terminal-pane-context-menu.ts:152-166`
  (`onCopy`) — awaits the write but never reads it back.
- `src/renderer/src/components/terminal-pane/use-terminal-pane-lifecycle.ts:1129-1135`
  (copy-on-select) — same swallowed `.catch`.

The main-process handler
`src/main/window/clipboard-ipc-handlers.ts:143-146` (`clipboard:writeText`)
calls `clipboard.writeText(...)`, which returns `void`; there is no read-back
confirmation at any layer.

## Automated evidence

A test that imports the **real** keyboard handler, dispatches the copy chord,
and pins the buggy behavior (write rejects → keystroke still consumed, rejection
swallowed, clipboard never read back to verify):

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/components/terminal-pane/repro-5611-copy-success-unverified.test.ts
```

Result on the current tree: **1 passed** — the handler calls
`writeClipboardText('selected copilot text')`, sets `event.defaultPrevented`,
produces zero unhandled rejections, and never calls `readClipboardText`.

## Manual repro (from the issue)

1. Select text in an Orca terminal/agent pane (Copilot conversation history).
2. Press `Ctrl+C` / `Ctrl+Shift+C` (or right-click copy).
3. Orca reports success (status flips to "Copied to clipboard").
4. Paste into another app — the clipboard is unchanged/empty.

## Fix direction

Centralize terminal-selection copy into one helper that: reads the xterm
selection, fails/indicates when it is empty, writes via the clipboard IPC, reads
the clipboard back to confirm the written text matches, and only then clears the
selection or signals success (and surfaces an error toast on failure instead of
swallowing it).
