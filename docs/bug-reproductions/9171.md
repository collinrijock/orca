# Issue #9171 — Wrong PR diffs displayed when checking out the default branch

**Status:** REPRODUCED

## Summary
When a repository's default branch (`master`) is checked out, Orca surfaces a
stale, previously-closed pull request (its diffs in Source Control and its status
in the Checks tab). The PR shown is one whose HEAD ref happened to be the default
branch — e.g. a PR someone accidentally opened *from* `master` and later closed.
Orca should attach no PR when sitting on the default branch.

## Root cause
`getPRForBranchOutcome` resolves the PR for a checkout purely by head-branch name
and has no awareness of the repo's default branch. For a `master` checkout it runs
a head-branch PR lookup:

- `src/main/github/client.ts:2380` `getRestPRForBranch` →
  `GET /repos/{owner}/{repo}/pulls?head={owner}:master&state=all&per_page=1`
- `src/main/github/client.ts:2396` `getFallbackPRListForBranch` →
  `gh pr list --head master --state all --limit 1`

Both queries use `state=all`, so a **closed** PR whose head ref is `master` is
returned (`list[0]`). The result flows through `lookupPRByBranchName`
(`src/main/github/client.ts:2733`) into `getPRForBranchOutcome`
(`src/main/github/client.ts:2962`). The only "hide" guard is
`isMergedImplicitPR` (`client.ts:2183`), which fires **only for merged** PRs — a
*closed* (unmerged) PR is not filtered, so it is returned as
`{ kind: 'found', pr }` and attached to the default-branch checkout.

There is no check that the branch being looked up is the repository's default
branch (where an implicit head-branch PR association is meaningless).

## Automated evidence
Unit test imports the real `getPRForBranch` / `getPRForBranchOutcome` and pins the
buggy result (test passes today because the stale closed PR is returned):

```
pnpm exec vitest run --config config/vitest.config.ts src/main/github/repro-9171-default-branch-stale-pr.test.ts
```

Result: 2 passed. The assertions marked `BUG` encode the wrong behavior; the
adjacent `CORRECT` comments state the intended result (`pr` should be `null` /
`outcome.kind === 'no-pr'`).

## Manual repro (app steps)
1. In a GitHub repo whose default branch is `master`, ensure at least one closed
   PR exists whose head branch was `master`.
2. Open the repo's default-branch checkout in Orca.
3. Observe the Source Control and Checks tabs show that closed PR's diffs/checks
   instead of nothing.

## Fix direction
Skip the implicit head-branch PR lookup (or discard its result) when the branch
equals the repository's default branch and there is no explicitly linked/fallback
PR number. Alternatively, exclude `closed` (unmerged) PRs from the implicit
head-branch association the same way merged PRs are hidden via
`isMergedImplicitPR`, since an implicit branch match to a non-open PR is stale
context. The default-branch guard is the more targeted fix and preserves the
"no PR on the trunk" expectation stated in the issue.
