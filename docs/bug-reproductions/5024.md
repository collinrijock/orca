# Issue #5024 — Purple File Names not Clickable

**Status:** PARTIAL (root cause proven at the code level with a deterministic unit test; the final "link is dropped" step happens in the live app's filesystem-existence probe)

## Summary

Claude Code (opus) prints file names in purple. Some are cmd-clickable (open the
file), others — visually identical purple text — are not, which is confusing.
Orca's terminal file-link detection never looks at the terminal cell color or
the bold attribute at all. Whether a mention becomes a clickable link is decided
purely by (1) matching path-shaped text and (2) that path resolving to an
existing file **relative to a single cwd** (the pane's startup / link cwd), or
being a known worktree root. "Rendered purple" therefore carries no signal about
clickability, and legitimate file mentions are silently dropped whenever they do
not resolve against that one cwd.

## Root cause

- Clickability gate — a detected path is only offered as a link if
  `window.api.shell.pathExists()` returns true for the path resolved against the
  pane's single cwd:
  `src/renderer/src/components/terminal-pane/terminal-link-handlers.ts:163-174`
  (`createFilePathLinkProvider`). No color/bold input anywhere in this provider.
- Single-cwd anchoring — relative and bare mentions are joined onto exactly one
  cwd: `src/renderer/src/lib/explicit-file-link-target.ts:64-72`
  (`resolveExplicitFileLinkTargetPath` → `joinAbsolutePath(cwd, pathText)`).
  A bare filename Claude mentions in prose (`terminal-links.ts`) anchors to
  `<cwd>/terminal-links.ts` even though the real file is nested, and a
  repo-root-relative path anchors under the pane cwd when that cwd is a subdir.
- Bare-filename / word detection also requires existence to avoid noise
  (`src/renderer/src/lib/terminal-links.ts:429-453`,
  `looksLikeFilename` at :155-169), so anything that fails the cwd probe never
  underlines on hover and never opens on cmd+click.
- Minor detection defect: paths Claude abbreviates with an ellipsis
  (`src/…/lib/terminal-links.ts`) drop the pre-`…` prefix and mis-detect as an
  absolute `/lib/terminal-links.ts` rooted at the filesystem root.

This matches maintainer @Jinwoo-H's comment on the issue: the failing cases are
exactly "the file did not exist relative to the agent terminal's cwd."

## Automated evidence

Deterministic unit test that imports the real detection + resolution modules
(no fs, no mocks) and pins the buggy/gap behavior — it passes on the current
tree:

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/lib/repro-5024-purple-file-not-clickable.test.ts
```

Result: `Test Files 1 passed (1) / Tests 4 passed (4)`. The test shows:
- color/bold is never consulted;
- a bare filename mention anchors to the cwd root (`/home/user/repo/terminal-links.ts`), not where the file lives;
- a repo-root-relative path anchors under a subdir cwd (`.../packages/app/src/renderer/App.tsx`);
- an ellipsis-truncated path mis-detects as `/lib/terminal-links.ts`.

Each of these resolved targets does not exist, so the live provider's existence
gate drops them → purple text that is not clickable.

## Manual repro (live app)

1. Open a terminal whose cwd is a repo subdirectory, or run Claude Code (opus)
   in a worktree.
2. Have the agent print a file it changed as a bare name or repo-root-relative
   path that is not directly under the terminal cwd (e.g. prose "updated
   `terminal-links.ts`", or a summary list).
3. Hover / cmd+click the purple name: no underline, nothing opens — while a
   sibling path that happens to resolve under the cwd (or a `~/…` absolute path)
   is clickable.

## Fix direction

- Try resolving agent-mentioned relative/bare paths against the worktree root
  (and known worktree roots), not only the pane cwd, before giving up.
- Optionally fall back to a repo-relative filename search when a single-cwd join
  misses, so bare mentions of existing files become clickable.
- Reject obviously-elided (`…`) path fragments instead of emitting a bogus
  absolute-root link.
- Consider making the color/clickability relationship less surprising (only
  files Orca can actually open should read as links).
