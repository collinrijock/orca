# #7623 — Headless PR read-path runtime routing clobber (follow-up to #7590)

**Status:** REPRODUCED (read-path siblings; item 1 already fixed by #7739)

## Summary

The read-only assignee/label/reviewer panels in `PullRequestPage.tsx` and
`GitHubItemDialog.tsx` resolve their runtime routing with an **unconditional
spread** of `getTaskSourceRuntimeSettings(sourceContext)` over the repo-owner
settings. Because `getTaskSourceRuntimeSettings` always returns an object whose
`activeRuntimeEnvironmentId` key is **present** (value `null` for a local source
host), spreading it unconditionally **clobbers a valid repo-owner runtime id
with `null`** whenever a runtime-owned/headless repo is viewed through a local
GitHub source. The metadata reads then route to the local host instead of the
owner runtime — the same root cause as the #6957/#7590 merge bug, on the read
path.

The merge/mutation path (item 1 of the issue) was fixed by #7739: `PRActionsPanel`
in `PullRequestPage.tsx` now routes through `getGitHubMutationRoutingSettings`.
The read-path siblings listed in the issue remain unfixed.

## Root cause

`src/shared/task-source-context.ts:113-120` — `getTaskSourceRuntimeSettings`
always returns `{ activeRuntimeEnvironmentId: <id-or-null> }` (key always present).

Read-path sites still spread it unconditionally over `repoOwnerSettings`:

- `src/renderer/src/components/PullRequestPage.tsx:545` (`PRAssigneesPanel`),
  `:780` (`PRReviewersPanel`), `:3236` (`ConversationTab`), `:5823` (`GHEditSection`)
- `src/renderer/src/components/GitHubItemDialog.tsx:498` (`PRAssigneesPanel`),
  `:736` (`PRReviewersPanel`), `:5610` (`GHEditSection`)

```ts
const sourceSettings =
  sourceContext?.provider === 'github'
    ? ({ ...repoOwnerSettings, ...getTaskSourceRuntimeSettings(sourceContext) } as ...)
    : repoOwnerSettings
```

For a runtime-owned repo (`repoOwnerSettings.activeRuntimeEnvironmentId = 'env-owner'`)
viewed via a local GitHub source, the spread produces `{ activeRuntimeEnvironmentId: null }`
→ `getActiveRuntimeTarget` returns `{ kind: 'local' }`.

The already-correct helper is
`src/renderer/src/lib/github-source-runtime-context.ts:36` —
`getGitHubMutationRoutingSettings`, which is repo-owner-first and only lets a
*runtime* source override (`sourceHost?.environmentId ?? getExplicitRuntimeOwnerEnvironmentId(...)`).

## Automated evidence

Re-run command:

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/components/repro-7623-readpath-clobber.test.ts
```

Result: 5 passed. The test imports the real product helpers
(`getTaskSourceRuntimeSettings`, `getSettingsForRepoRuntimeOwner`,
`getGitHubMutationRoutingSettings`, `getActiveRuntimeTarget`), reconstructs the
runtime-owned-repo + local-source topology, and pins the buggy result:

- `readPathSourceSettings()` (the exact product spread) → `activeRuntimeEnvironmentId: null`
  → `getActiveRuntimeTarget` = `{ kind: 'local' }` (WRONG).
- `getGitHubMutationRoutingSettings(...)` → `activeRuntimeEnvironmentId: 'env-owner'`
  → `{ kind: 'environment', environmentId: 'env-owner' }` (CORRECT).

A structural guard also asserts the unconditional spread is still present at the
read-path sites in the current tree, so any fix flips the test.

## Manual repro (app)

1. Pair a headless/runtime-owned repo (web/mobile/remote client) so the repo's
   execution host is a runtime environment.
2. Open a PR for that repo from the standalone PR page (or the GitHub item dialog)
   through a **local** GitHub source view.
3. Open the assignees/reviewers/labels rail. The metadata reads route to the
   local host instead of the owner runtime → stale/empty results.

## Fix direction

Route every read-path site through the repo-owner-aware helper (as the issue
suggests): extract/reuse a single `resolveGitHubSourceSettings(repoOwnerSettings,
sourceContext)` that mirrors `getGitHubMutationRoutingSettings` (repo-owner base,
guarded runtime-source override — only override when
`getTaskSourceRuntimeSettings(sourceContext).activeRuntimeEnvironmentId` is truthy).
Replace all `{ ...repoOwnerSettings, ...getTaskSourceRuntimeSettings(sourceContext) }`
spreads with it.
