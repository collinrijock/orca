# Issue #8259 — Orca doesn't recognize a Cursor agent session in the sidebar (Windows)

**Status:** REPRODUCED

## Summary
On Windows, launching Cursor Agent from a normal PowerShell terminal inside an
Orca worktree runs and responds correctly, but Orca shows no Cursor icon or
agent status in the sidebar. The reason is that Cursor's Windows launcher does
not run a process literally named `cursor-agent`; it runs the bundled Node
runtime (`node.exe`) against a versioned script whose entrypoint basename is
`index.js` under a `cursor-agent\<version>\` directory. Orca's foreground
process-name / command-line matcher fails to recognize this wrapped launch.

## Root cause
`src/shared/agent-process-recognition.ts`

- `recognizeAgentProcessFromCommandLine()` (line 284) tokenizes the command line.
  For a `node.exe ...\cursor-agent\...\index.js` argv, the first token normalizes
  to `node`, so it falls through to `findInterpreterEntrypointToken()` and picks
  the script path as the entrypoint.
- The entrypoint is then resolved via `recognizeAgentProcess(entrypoint)` or
  `recognizeNodeScriptEntrypoint(entrypoint)` (line 312).
  - `recognizeAgentProcess` normalizes the basename to `index.js` → no match.
  - `recognizeNodeScriptEntrypoint` (line 199) strips the `.js` to `index` and
    looks it up in `NODE_PACKAGE_SCRIPT_ENTRYPOINTS` (line 51), which only
    contains markers for **codex** and **gemini** — not cursor.
- Cursor's `cursor-agent` config (`src/shared/tui-agent-config.ts:272`) only
  declares `expectedProcess: 'cursor-agent'`, which never matches the `node` /
  `index.js` wrapper the Windows launcher actually spawns.

Result: `recognizeAgentProcessFromCommandLine(...)` returns `null` for the real
Windows Cursor command line. That value is what
`src/main/providers/windows-agent-foreground-process.ts` (lines 92-93, 113-114)
uses to identify the foreground agent, so no Cursor identity/status is surfaced.

## Automated evidence
A unit test imports the real product module and pins the buggy behavior (it
passes today by asserting detection returns `null`):

```
pnpm exec vitest run --config config/vitest.config.ts src/shared/repro-8259-cursor-node-wrapper.test.ts
```

Result: 3 passed. The baseline case (a bare `cursor-agent` process) is
recognized, while the versioned `node.exe ...\cursor-agent\...\index.js` wrapper
and the `node <path>\index.js` argv form both return `null` (the bug).

## Manual repro (Windows)
1. On Windows, open a PowerShell terminal in an Orca worktree.
2. Run `cursor-agent` (the installed Cursor CLI wrapper).
3. Observe the agent runs and responds, but the Orca sidebar shows no Cursor
   icon or agent status for that terminal.

## Fix direction
Teach the entrypoint matcher to recognize Cursor's Node wrapper, e.g.:
- Add a `NODE_PACKAGE_SCRIPT_ENTRYPOINTS` (or an analogous directory-marker)
  entry that matches a `cursor-agent/<version>/index.js` path and maps to the
  `cursor` agent, and/or
- Recognize the `cursor-agent` directory segment in the entrypoint path
  (basename `index.js` is not distinctive on its own, so match on the containing
  `cursor-agent` directory).
Also make the hook-based fallback a reliable identity source, and add Windows
regression coverage. Keep matching path-anchored to avoid substring false
positives (the existing `tokenLooksExecutable` guard exists for this reason).
