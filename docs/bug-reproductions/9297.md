## Summary

Orca's local agent detection resolves each CLI candidate by **spawning the OS
`where`/`which` binary as a child process â€” one subprocess per candidate**
(~30 candidates; 34 probe commands on win32). Under corporate
privilege-management / application-control software that intercepts each
`where.exe` spawn behind an interactive approval, those dozens of gated spawns
stall Orca's main process, matching the ~10sâ€“5min startup delays in the report.
This is the Windows counterpart of the macOS Endpoint-Security hang in #5657.

Nuance vs. the triage note: the current code is **already parallel** (a single
`Promise.all` over all probes in `detectInstalledAgents`) and each spawn has a
**5s timeout** (`PREFLIGHT_COMMAND_TIMEOUT_MS`). So the issue's "serial / no
timeout" theory is not literally accurate. What *is* still true â€” and is the
issue's primary suggested fix ("avoid shelling out to `where.exe`/`which`;
resolve via Node `fs`") â€” is that detection spawns a `where.exe` subprocess per
candidate at all. When the gating tool serializes its approval prompts (one at
a time), 34 pending spawns still queue up and block startup, and the 5s
per-spawn timeout does not bound the total.

## Root cause

- `src/main/ipc/preflight-command-exec.ts:79` â€” `isCommandOnPath()` resolves a
  command by spawning the OS finder: `const finder = process.platform === 'win32' ? 'where' : 'which'` then `await execLocalPreflightCommand(finder, [command])` (line 83 / 96). No fs-based PATH lookup path exists.
- `src/main/ipc/preflight.ts:101-106` â€” `detectInstalledAgents()` calls
  `isCommandOnPath(cmd)` for **every** probe command, so one `where.exe`/`which`
  subprocess is spawned per agent candidate.
- Candidate list: `KNOWN_TUI_AGENT_DETECTION_COMMANDS`
  (`src/main/ipc/tui-agent-detection-commands.ts`) derived from
  `TUI_AGENT_CONFIG` (~30 agents) â†’ 34 probe commands on win32.

## Automated evidence

## Fix direction

Replace subprocess-based PATH resolution with fs-based resolution: split
`process.env.PATH` (respecting `PATHEXT` on Windows) and check file existence
per candidate â€” zero subprocess spawns. If subprocess probing must stay for
some cases, additionally bound the *aggregate* detection (single short overall
budget) and cache the result, so a gated/serialized approval queue cannot
stall startup. Detection also already runs behind the renderer-driven
`preflight:detectAgents` IPC; consider not blocking window creation on it.
