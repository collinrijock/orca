# Issue #7331 — PR details fail to load for fork repositories (`getOwnerRepo` only checks `origin`)

**Status:** REPRODUCED

## Summary

On a fork checkout (`origin` → personal fork, `upstream` → parent repo), opening a
PR in Orca shows "Unable to load details for this GitHub item." The PR-detail
lookup resolves the repo via `getOwnerRepo()`, which consults **only** the
`origin` remote. So Orca runs `gh pr view <N> --repo <fork>`, and since PRs live
on the parent repo, GitHub returns "Could not resolve to a PullRequest". Issue
details work because `getIssueOwnerRepo()` checks `upstream` first.

## Root cause

`src/main/github/github-repository-identity.ts:206` (`getOwnerRepo`):

```ts
export async function getOwnerRepo(repoPath, connectionId, localGitOptions) {
  return getOwnerRepoForRemote(repoPath, 'origin', connectionId, localGitOptions) // origin only
}
```

vs. `getIssueOwnerRepo` at `github-repository-identity.ts:214`, which tries
`upstream` first and only falls back to `origin`.

`getOwnerRepo()` is the resolver used by every `gh pr view` PR-detail path in
`src/main/github/client.ts` (e.g. lines 3613, 3886, 3979, 4119, 4474, 4516,
4578, 4624, 4831, 4869, 4902) when no explicit `prRepo` is supplied. On a fork
checkout it targets the fork, so the PR-detail fetch fails.

## Automated evidence

Unit test importing the real product module and pinning the buggy divergence
between the two resolvers:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/github/repro-7331-fork-pr-owner-repo.test.ts
```

Result: 3 passed. The mocked git runner returns `origin → fsdwen/orca` and
`upstream → stablyai/orca`. `getOwnerRepo()` returns `{owner:'fsdwen',repo:'orca'}`
(the fork — the bug), while `getIssueOwnerRepo()` returns
`{owner:'stablyai',repo:'orca'}` (the parent). The third test asserts the two
resolvers disagree for the same checkout, which is exactly the divergence #7331
reports.

## Manual repro

1. Fork `stablyai/orca`, clone the fork; add `upstream` → `stablyai/orca`.
2. Open the repo in Orca, open a PR from the PR list.
3. PR details fail: "Unable to load details for this GitHub item."

## Fix direction

Make `getOwnerRepo()` prefer `upstream` then fall back to `origin`, matching
`getIssueOwnerRepo()` — or route the PR-detail paths through
`resolvePRRepositoryCandidates()` (already present) which returns both
upstream+origin candidates. Note the fix must preserve fork-push semantics where
the fork *is* the correct head repo for creating PRs.
