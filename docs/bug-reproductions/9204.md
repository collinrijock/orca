# Issue #9204 — Undo fails in Diff View after saving; manual revert forces edited code to reappear

**Status:** REPRODUCED (root cause proven at the logic level with an executed unit test; full paint requires the live app but the buggy model-identity math and Monaco reuse contract are exercised directly)

## Summary
In the editable Compare/Diff view (an unstaged single-file diff), the modified pane's
Monaco model identity is a pure function of the git-fetched modified content. Saving
overwrites that fetched content with the just-saved text, so the model URI (and the
React `key`) rotate on every save. Two symptoms follow:

1. **Undo breaks after save.** The rotation remounts the `DiffEditor` and creates a
   brand-new Monaco model, discarding the undo stack — so Cmd/Ctrl+Z no-ops.
2. **Manual revert resurrects the edited code.** Content signatures repeat, so
   reverting the pane back to the original text rotates the URI back to a *previously
   created* model. `keepCurrentModifiedModel` keeps that old model alive with its
   stale (edited) content, and `@monaco-editor/react` reuses an existing model at a
   URI **as-is**, ignoring the `modified` prop — so the earlier edit reappears and
   overwrites the manual restoration.

## Root cause
- `src/renderer/src/components/editor/EditorContent.tsx:963` — `modifiedModelKey`
  embeds `getDiffContentSignature(dc.modifiedContent)`.
- `src/renderer/src/components/editor/EditorContent.tsx:966` — the `DiffViewer`
  React `key` also embeds `getDiffContentSignature(dc.modifiedContent)`, so any change
  to `dc.modifiedContent` fully remounts the diff editor.
- `src/renderer/src/components/editor/useEditorPanelExternalContentEvents.ts:91-97` —
  on `ORCA_EDITOR_FILE_SAVED_EVENT`, `dc.modifiedContent` is overwritten with the saved
  text. This is what rotates the signature on save.
- `src/renderer/src/components/editor/DiffViewer.tsx:456-458` — `modifiedModelPath`
  is derived from that key and `keepCurrentModifiedModel` is set, so old models are
  never disposed and can be resurrected.
- `src/renderer/src/components/editor/diff-monaco-model-disposal.ts`
  (`getDiffViewerMonacoModelPaths`) + `diff-content-signature.ts`
  (`getDiffContentSignature`) — the model path is a deterministic function of content,
  so identical content ⇒ identical URI (the collision).
- Third-party contract: `@monaco-editor/react@4.7.0` `DiffEditor`
  (`node_modules/@monaco-editor/react/dist/index.js`) resolves the modified model via
  `getModel(path) || createModel(value, path)` — an existing model at `path` is reused
  **without** applying the `modified` prop, which is the resurrection mechanism.

The comment at `diff-content-signature.ts` intends the signature to rotate only when
git-fetched blob content changes (not on keystrokes); the defect is that a *save* is
treated exactly like a git-blob change, rotating identity mid-edit-session and then
colliding on revert.


Result: `Test Files 1 passed`, `Tests 3 passed`. The three tests pin: (a) the URI
rotates on save and collides on revert, (b) SYMPTOM 1 — a new model (fresh undo stack)
after save, (c) SYMPTOM 2 — `visibleContent()` returns the stale edited text after a
manual revert + save. Assertions encoding the bug are marked `BUG:` in the test.

## Manual repro (live app, macOS)
1. Open a modified file in Diff View (Compare mode, unstaged).
2. Edit the right-hand (modified) pane.
3. Save (Cmd+S).
4. Press Cmd+Z — nothing happens (undo stack was reset by the save-triggered remount).
5. Manually retype the original text and save again — the previously edited text snaps
   back, overwriting the manual restoration.

## Fix direction
- Do not rotate the modified-side model identity on the user's own save. Key the
  modified model on stable tab identity (+ an explicit reload nonce for *external* git
  changes) rather than on `getDiffContentSignature(dc.modifiedContent)`, so a save keeps
  the same model and undo stack. The original-side may still rotate on HEAD moves.
- Alternatively, when the modified model URI must change, dispose/reset the outgoing
  model (or force `setValue` of the `modified` prop on reuse) so a resurrected URI can
  never surface stale content.
