# Issue #7323 ‚Äî Injected Claude Code hooks have no timeout (60s stall)

**Status:** NOT REPRODUCED (already fixed in current tree)

## Summary

The issue reports that Orca injects managed Claude Code hooks into
`~/.claude/settings.json` **without a `timeout` field**, so Claude Code's 60s
default applies and every prompt / tool call stalls for up to 60s when Orca or
its relay isn't running (Windows-visible, but the config is the same on all OSes).

On the current tree this does **not** reproduce: every injected Claude hook entry
ships an explicit numeric `timeout`, and the hook transport additionally
fast-fails a dead relay in ~1.5s. The 60s-default scenario is closed.

## Root cause / current behavior

The injection path is `applyManagedHooks` in
`src/main/claude/hook-settings.ts:95`, which builds each hook command through
`buildManagedCommandHook` in
`src/main/agent-hooks/installer-utils.ts:51`:

```ts
export function buildManagedCommandHook(
  command: string,
  timeout = MANAGED_HOOK_TIMEOUT_SECONDS   // = 10  (installer-utils.ts:46)
): HookCommandConfig {
  return { type: 'command', command, timeout }
}
```

So every entry across all 11 `CLAUDE_EVENTS` (`hook-settings.ts:29`) is emitted
as `{ type: 'command', command, timeout: 10 }` ‚Äî an explicit 10s host-level
backstop, far below the 60s default the issue reported. Independently, the shell
wrapper's curl uses `--connect-timeout 0.5 --max-time 1.5`
(`installer-utils.ts:207,230`), so when the relay is down the hook returns in
~1.5s rather than blocking.

This was added by commit `e03a1cf769` / PR #6148 ("Prevent managed agent hooks
from hanging"), which post-dates the older `agent-flow/hook.js` architecture the
issue references.

Residual gap vs. the issue's ask: the issue requests a 1‚Äì3s explicit timeout;
the injected value is 10s. Because the curl transport already bounds a dead relay
at 1.5s, the user-facing stall is ~1.5s, not 10s and never 60s ‚Äî so the reported
symptom no longer occurs. Tightening the 10s backstop would be a hardening
follow-up, not a fix for the reported 60s stall.

## Fix direction

If a stricter bound is desired (issue asks 1‚Äì3s), lower the injected per-hook
`timeout` toward the ~1.5s curl transport budget in
`src/main/agent-hooks/installer-utils.ts` (`MANAGED_HOOK_TIMEOUT_SECONDS`). No
change is required to close the reported 60s-stall symptom.
