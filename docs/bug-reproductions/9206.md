# Issue #9206 — ORCA_ROOT_PATH and ORCA_WORKTREE_PATH do not exist in the WSL setup script

**Status:** REPRODUCED

## Summary
On Windows + WSL, the generated `setup-runner.sh` runs inside `wsl.exe`. The
setup env vars (`ORCA_ROOT_PATH`, `ORCA_WORKTREE_PATH`, `ORCA_WORKSPACE_NAME`,
plus the `CONDUCTOR_ROOT_PATH` / `GHOSTX_ROOT_PATH` compat vars) are placed on
the Windows-side PTY spawn env but are never registered in `WSLENV`. `wsl.exe`
only imports env vars whose names are listed in `WSLENV`, so inside the Linux
bash these variables are empty. The user's script
`cp "$ORCA_ROOT_PATH/.env" "$ORCA_WORKTREE_PATH/.env"` therefore expands to
`cp "/.env" "/.env"`, which fails with `cp: cannot stat '/.env'`.

## Root cause
- The setup env vars are generated correctly and even value-translated to Linux
  paths for WSL: `src/main/hooks.ts:343` (`getSetupEnvVars`), `hooks.ts:458`
  (`getSetupRunnerEnvVars`), and `hooks.ts:555-559` (path translation).
- But crossing into the WSL guest requires WSLENV registration, and the sole
  WSLENV gatekeeper on the WSL spawn path — `addOrcaWslInteropEnv`
  (`src/main/pty/wsl-orca-env.ts:17`, called at
  `src/main/daemon/pty-subprocess.ts:759` and mirrored in
  `src/main/providers/local-pty-provider.ts`) — has a fixed passthrough
  allowlist (`wsl-orca-env.ts:24-41`) that does **not** include `ORCA_ROOT_PATH`,
  `ORCA_WORKTREE_PATH`, `ORCA_WORKSPACE_NAME`, `CONDUCTOR_ROOT_PATH`, or
  `GHOSTX_ROOT_PATH`. No `addWslEnvKeys(...)` call registers them either.

Net effect: the values are set on the Windows-side process but never imported by
`wsl.exe`, so `$ORCA_ROOT_PATH` / `$ORCA_WORKTREE_PATH` are empty inside the
setup script.

## Automated evidence
Unit test that imports the real `getSetupRunnerEnvVars` (hooks.ts) and the real
`addOrcaWslInteropEnv` (wsl-orca-env.ts) and pins the buggy behavior — WSLENV
omits the setup vars while forwarding an allow-listed var:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/repro-9206-wsl-setup-env-not-crossing.test.ts
```

Result: `Test Files 1 passed`, `Tests 2 passed`. The test asserts the setup vars
are absent from `WSLENV` (the current, wrong behavior) and documents which
assertions must flip once fixed.

## Manual repro (app steps)
1. On Windows, add a WSL-backed project whose worktrees live on the WSL filesystem.
2. Set a repo setup script: `cp "$ORCA_ROOT_PATH/.env" "$ORCA_WORKTREE_PATH/.env"`.
3. Create a worktree; the setup terminal runs `bash .../setup-runner.sh` in WSL.
4. Observe `cp: cannot stat '/.env': No such file or directory` because both vars
   are empty inside WSL.

## Fix direction
Register the setup env var names in `WSLENV` on the WSL spawn path — add
`ORCA_ROOT_PATH`, `ORCA_WORKTREE_PATH`, `ORCA_WORKSPACE_NAME`,
`CONDUCTOR_ROOT_PATH`, `GHOSTX_ROOT_PATH` to the `passthroughEntries` allowlist
in `addOrcaWslInteropEnv` (`src/main/pty/wsl-orca-env.ts`), each with the `p`
flag if a Windows path is passed or `u` if the value is already a Linux path
(values are Linux-translated at `hooks.ts:555-559`, so `u` is appropriate). This
matches the existing treatment of other Orca vars that must cross into WSL.

## Key files
- `src/main/pty/wsl-orca-env.ts` (WSLENV passthrough allowlist — missing the setup vars)
- `src/main/hooks.ts` (`getSetupEnvVars` / `getSetupRunnerEnvVars`, value translation)
- `src/main/daemon/pty-subprocess.ts:759` and `src/main/providers/local-pty-provider.ts` (WSL spawn call sites)
- `src/main/repro-9206-wsl-setup-env-not-crossing.test.ts` (this repro)
