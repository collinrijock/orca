# Issue #7797: Agents running inside a user's tmux session are not detected/shown as running

**Status:** REPRODUCED

## Summary

When a user runs `tmux` inside an Orca pane and then starts an agent (`claude`,
`codex`, …) in a tmux window, Orca never shows the agent as live/running. The
pane reads as `tmux`. The file-based Agent Session History (AI Vault) still
surfaces the transcript by `cwd`; only the live/running indicator is missing.

## Root cause

Live foreground detection walks the process tree **downward from the pane's
shell pid** and only inspects that subtree:

- `src/main/providers/agent-foreground-process.ts:20` — `collectDescendants(rows, rootPid)`
- `src/main/providers/agent-foreground-process.ts:108` — `resolveAgentForegroundProcessFromPs` calls `collectDescendants(rows, shellPid)`

tmux double-forks its **server** and reparents it to pid 1 (init). The process
the pane shell actually launched is the tmux **client**; every pane process —
including the agent — is a child of the reparented **server**, not of the pane
shell. So the descendant walk from the shell pid reaches only the tmux client
(not a recognized agent) and `resolveAgentForegroundProcessFromPs` returns the
`tmux` fallback. The OSC-133/9999 status path also misses it because tmux does
not forward those sequences to the outer terminal by default.

The relay/SSH twin (`src/relay/pty-shell-utils.ts`) has the same downward-only
walk, so remote sessions are affected identically.

## Automated evidence

A unit test imports the real `resolveAgentForegroundProcess` and feeds it a
`ps` snapshot modelling the tmux topology (shell pid 100 → tmux client 101;
tmux server 200 reparented to pid 1 → `claude` 201). It pins the buggy result
and includes a control proving the agent IS recognizable when walked from the
server pid — i.e. it is merely unreachable from the shell pid.

Re-run:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/providers/repro-7797-tmux-agent-detection.test.ts
```

Result: 2 passed. The `BUG:` test asserts `resolveAgentForegroundProcess(100, 'tmux')`
returns `'tmux'` (wrong); the `CONTROL:` test shows the same table yields
`'claude'` when the walk starts from the tmux server pid 200.

## Manual repro (app steps)

1. Open an Orca pane, run `tmux`.
2. Inside tmux, run `claude` (or `codex`).
3. The pane / worktree / project view never marks the agent as running; the
   pane is reported as `tmux`.

## Fix direction

When the shell subtree contains a tmux client, hop to that client's active pane
pid and re-run the same recognition walk from there:

1. Detect a tmux client among the descendants.
2. `tmux [-L/-S <sock>] list-clients -F '#{client_pid} #{pane_pid}'` to map our
   client pid to its active pane pid.
3. Re-run the existing descendant recognition from that pane pid (and also
   consider the pane's own top process for `tmux new-session claude`).

Best-effort (any tmux failure falls through unchanged), POSIX-only, mirrored in
the local and relay/SSH paths, with a short TTL cache to avoid forking `tmux`
on every foreground poll.
