# #7209 — SSH host terminal size does not match the pane (locked to host pane dimensions) on macOS

**Status:** REPRODUCED (root cause pinned by an executed unit test against the real relay `PtyHandler`)

## Summary

When a desktop pane connects to an existing SSH-host PTY, the remote terminal
is sized to whatever dimensions the PTY had when it was first spawned (or last
sized by a previous client) rather than the local pane's current size. Content
is clipped / wraps wrong / leaves dead margin, and resizing the window does not
reliably fix it.

The relay's `pty.attach` handler receives the attaching client's pane
`cols`/`rows` but silently drops them — it never resizes the backing PTY on
attach. Every reattach path (app restart, SSH reconnect, tab reveal) therefore
pins the remote PTY to the *previous* size ("host pane dimensions").

## Root cause

`src/relay/pty-handler.ts` — `attach()` (defined at ~line 940). The method
proves the backing shell is alive and replays the buffered output, but it never
calls `managed.pty.resize(cols, rows)`, even though `params.cols` / `params.rows`
are present in the request:

- The desktop SSH provider explicitly sends the local pane size on reattach:
  `src/main/providers/ssh-pty-provider.ts` `spawn({ sessionId })` issues
  `this.mux.request('pty.attach', { id, cols: opts.cols, rows: opts.rows, ... })`
  (lines ~116–123).
- The relay handler consumes only `id`, `expectedPaneKey`, `expectedTabId`, and
  `suppressReplayNotification` — the `cols`/`rows` are ignored.

Contrast with `resize()` in the same file (line ~1024), which does
`managed.pty.resize(cols, rows)`. Attach has no equivalent, so the client's
declared viewport is lost until some *later* desktop-driven resize happens to
fire — which the issue notes is unreliable.

The renderer's own drift-correction (`pty-size-reassertion.ts`) cannot rescue
this for SSH: its readback (`window.api.pty.getSize`) falls through to the
requested-size cache for SSH PTYs (the SSH provider has no `getAppliedSize`, see
`src/main/ipc/pty.ts` `pty:getSize`, ~line 5311), so it never observes the
remote/local size divergence and never re-forwards.

## Automated evidence

The test imports the real relay `PtyHandler`, spawns a PTY at 80x24, then
attaches with a different local pane size (200x50) exactly as the desktop SSH
provider does, and asserts the PTY's `resize()` is never called — pinning the
bug. Correct behavior (commented in the test) is `resize(200, 50)` on attach.


## Manual repro (app steps)

1. Connect to an SSH host on macOS and open a terminal on it.
2. Get the relay PTY to a known size, then reattach from a differently-sized
   pane — e.g. restart Orca (or drop/restore the SSH connection), or resize the
   window while the tab is hidden and then reveal it.
3. The SSH terminal renders at the pre-existing size, not the current pane size:
   clipped edges / wrong wrapping / empty margin. Resizing does not reliably
   correct it.

## Fix direction

In `PtyHandler.attach()`, after proving liveness (and before/after replay),
apply the attaching client's viewport when provided:

```ts
const cols = Math.max(1, Math.min(500, Math.floor(Number(params.cols) || 0)))
const rows = Math.max(1, Math.min(500, Math.floor(Number(params.rows) || 0)))
if (cols > 0 && rows > 0) {
  managed.pty.resize(cols, rows)
}
```

This mirrors the bounds already used in `resize()` and makes attach "claim" the
local pane size, the same way the mobile / remote-desktop takeover flow resizes
the shared PTY (the "Restore desktop size" behavior referenced in #1900 / #4221).
Note `attachForReconnect` in `ssh-pty-provider.ts` currently sends no cols/rows,
so the reconnect path should also pass the pane size for the fix to cover it.
