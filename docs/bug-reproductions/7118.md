# Issue #7118: Opening or closing sidebar resets terminal scroll to top

**Status:** NOT REPRODUCED (fixed in current tree)

## Summary

Reported against v1.4.117.0 (2026-07-02): toggling the sidebar jumped the
terminal's scroll position all the way to the top, losing the user's scrolled-up
viewport. On the current tree the scroll position is preserved across a sidebar
toggle — the responsible refit path now captures the pinned viewport before
xterm's `fit()` and restores it afterward.

## What actually happens (verified in code)

1. Toggling the sidebar changes the terminal container's width.
2. The pane's `ResizeObserver` (`attachPaneFitResizeObserver`,
   `src/renderer/src/lib/pane-manager/pane-fit-resize-observer.ts:148`) fires and
   calls `requestStablePaneFit` → `safeFit`.
3. `safeFit` → `performSafeFit`
   (`src/renderer/src/lib/pane-manager/pane-fit.ts:73`). Because the width really
   changed, the proposed grid differs from the current grid, so it takes the
   real-fit branch. Before calling `pane.fitAddon.fit()` it runs
   `captureScrollForFit()` (line 83/113), snapshotting the pinned viewport via
   `captureTerminalStructuralScrollIntent` + `captureScrollState`.
4. xterm's `fit()` reflows and resets the native viewport to the top.
5. The `finally` block (lines 121-149) restores the captured scroll through
   `restoreScrollStateAfterFit` (`pane-scroll.ts:100`), marker-tracking the pinned
   content so the viewport lands back on the same line it was on.

The bug the user saw corresponds to step 4 with no step 3/5 — which is not the
current behavior.

## Root cause (of the original bug) and the fix

The pre-fix refit path called `fitAddon.fit()` without capturing/restoring the
viewport, so every width-driven reflow snapped scroll to the top. The
scroll-preserving refit was introduced in `eeb18da4bd` ("Fix terminal scroll
restore during resize", 2026-05-19) and later reworked to be transactional in
`69c14fadce` (#8674 "make viewport ownership transactional across output,
replay, and reflow", 2026-07-15). Both are ancestors of the current HEAD, so the
fix is present.

Note the original report is dated 2026-07-02, after the May fix but before
#8674; the current tree is post-#8674.

## Automated evidence

A repro test drives the REAL product refit path (`safeFit` + the real
`terminal-scroll-intent` machinery) for the exact sidebar-toggle scenario: a
user scrolled up to line 42 (pinned viewport), then a width-changing refit whose
`fit()` clobbers the viewport to 0. It asserts the viewport is restored to 42.

Re-run:

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/lib/pane-manager/repro-7118-sidebar-toggle-scroll.test.ts
```

Result: 1 passed — `scrollToLine(42)` is called and `viewportY` ends at 42. If
#7118 still reproduced, `viewportY` would be left at 0 (top) and `scrollToLine`
would never fire (that alternative assertion is noted in the test comments).

## Manual repro (app steps)

1. Open a terminal with enough scrollback to scroll.
2. Scroll up so the viewport is detached from the bottom.
3. Toggle the sidebar (open/close).
4. Expected on current tree: scroll position is retained. (Original bug:
   viewport jumps to the top.)

## Fix direction

No change needed — already fixed. If a regression surfaces, the guard lives in
`performSafeFit` (`pane-fit.ts`): ensure `captureScrollForFit()` runs on the
real-fit branch and `restoreScrollStateAfterFit` runs in the `finally`.
