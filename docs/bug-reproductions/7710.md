# Issue #7710 — OMP fresh launches can resume a previous session across new terminals/worktrees

**Status:** REPRODUCED

## Summary

When Orca spawns a *fresh* OMP terminal (new terminal or new worktree), the
command it builds is a bare `omp` (or `omp <prompt>`) with no session selector.
OMP auto-resumes its most-recent session when invoked with no explicit session
flag, so the "fresh" tab reattaches to the previous OMP session's transcript and
draft state instead of starting clean.

## Root cause

The OMP launch command is built by the shared TUI startup planners, which never
inject a fresh-session selector for OMP:

- `src/shared/tui-agent-config.ts:166-172` — OMP config: `launchCmd: 'omp'`,
  `promptInjectionMode: 'argv'`, `draftPromptEnvVar: 'ORCA_OMP_PREFILL'`. No
  fresh-session flag is associated with the agent.
- `src/shared/tui-agent-startup.ts:87-96` — promptless fresh launch returns
  `launchCommand: baseCommand.command`, i.e. exactly `omp`.
- `src/shared/tui-agent-startup.ts:100-112` — prompted fresh launch returns
  `omp <quoted prompt>` (still no session selector).
- `src/shared/tui-agent-startup.ts:298-308` — draft launch returns
  `omp; unset ORCA_OMP_PREFILL` (still no session selector).
- `src/shared/tui-agent-launch-command.ts:21-62` — `resolveAgentLaunchCommand`
  only appends user `agentArgs` / session-option picker flags; it has no notion
  of a per-launch fresh-session directory.

The codebase already knows `--no-session` is a valid Pi/OMP-family flag — it is
used for the headless commit-message agent
(`src/shared/commit-message-agent-spec.ts:445`) — but the interactive launch
path does not use any fresh-session selector.

Observed launch commands on the current tree (darwin):

| Path | launchCommand |
| --- | --- |
| empty-prompt fresh launch | `omp` |
| prompt fresh launch | `omp 'do a thing'` |
| draft launch | `omp; unset ORCA_OMP_PREFILL` (env `ORCA_OMP_PREFILL=draft text`) |

None carry `--session-dir` / `--no-session` / `--fork`, so OMP auto-resumes.

## Automated evidence

Unit test that imports the real product planners and pins the buggy behavior
(passes on the current tree while asserting the wrong result):

```
pnpm exec vitest run --config config/vitest.config.ts src/shared/repro-7710-omp-fresh-session.test.ts
```

Result: 4 passed. The buggy assertions are that each fresh-launch command is a
bare `omp` invocation with `SESSION_SELECTOR_RE.test(...) === false`. A fourth
assertion documents the invariant a fix must preserve: an explicit
user-supplied `--session-dir` is carried through untouched.

## Manual repro (from the issue)

1. Open an Orca worktree with an OMP terminal; leave a session with visible
   transcript/draft content.
2. Open another OMP terminal in a new worktree (or another terminal in the same
   worktree).
3. The new OMP terminal shows the previous session's transcript/prompt content.

## Fix direction

For Orca-created *fresh* OMP tabs (promptless, prompted, and draft paths), inject
an explicit fresh-session selector — e.g. a per-launch `--session-dir <fresh>`
(or `--no-session`) — carried all the way to the host spawn. Only inject it when
the user has NOT already supplied an explicit session selector
(`--resume`, `--continue`, `--session-dir`, `--no-session`, `--fork`) via
`agentArgs`, so explicit user resume/session intent is preserved. The natural
seam is `resolveAgentLaunchCommand` / the OMP branch of the startup planners in
`src/shared/tui-agent-startup.ts`.
