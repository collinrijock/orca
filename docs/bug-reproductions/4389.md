# Issue #4389 â€” Multiple orchestrators in a single workspace kill each other

**Status:** REPRODUCED

## Summary

Coordinator (orchestrator) ownership is tracked at **module/process scope**, not per
worktree. Running a second orchestrator for a *different* worktree in the same workspace
is rejected as "already running", and a stop request issued from one orchestrator's scope
tears down the other orchestrator's run (and its worker agents).

## Root cause

Two coupled defects in `src/main/runtime/rpc/methods/orchestration-gates.ts`:

1. **Global start gate.** `orchestration.run` gates on `db.getActiveCoordinatorRun()`
   (`orchestration-gates.ts:50-53`). That query has **no worktree predicate**:
   `src/main/runtime/orchestration/db.ts:1010-1016`
   ```sql
   SELECT * FROM coordinator_runs WHERE status = 'running' ORDER BY created_at DESC LIMIT 1
   ```
   The `worktree` selector passed to `orchestration.run` is handed to the `Coordinator`
   instance but is never persisted on the run row nor used for the active-run check, so a
   coordinator for `worktree:wt-b` collides with the already-running `worktree:wt-a` run.

2. **Global stop + module-global handle.** The active coordinator is stored in a single
   module-global `let activeCoordinator` (`orchestration-gates.ts:10`). `orchestration.runStop`
   accepts an **empty params schema** (`RunStopParams = z.object({})`,
   `orchestration-gates.ts:18`), resolves the same unscoped `getActiveCoordinatorRun()`, and
   calls `activeCoordinator.stop()` (`orchestration-gates.ts:88-102`). There is no way to
   scope a stop to a worktree, so a stop from B targets A's run.

## Automated evidence

Unit test imports the real `ORCHESTRATION_GATE_METHODS` handlers and the real
`OrchestrationDb`; only the `Coordinator` background loop is stubbed so the fire-and-forget
`executeLoop` cannot self-complete and race the assertions. The test PASSES today while
asserting the WRONG (buggy) results:

- Case 1: starting coordinator B for `worktree:wt-b` while A (`worktree:wt-a`) is active
  throws `Coordinator already running: <A run id>`.
- Case 2: `orchestration.runStop` has no worktree param (`{worktree:'wt-b'}` parses to `{}`),
  returns `{ runId: <A run id>, stopped: true }`, and calls `.stop()` on A's coordinator.

## Manual repro (app steps)

1. In one workspace, start orchestrator A scoped to worktree `wt-a`.
2. While A runs, start orchestrator B scoped to worktree `wt-b` â†’ fails with
   "Coordinator already running: <A>".
3. With A running, issue a run-stop from B's context â†’ A's run (and its workers) is stopped.

## Fix direction

- Scope active-run ownership by worktree: add a `worktree` column to `coordinator_runs`,
  persist it in `createCoordinatorRun`, and make `getActiveCoordinatorRun` accept/filter by
  worktree selector.
- Replace the single module-global `activeCoordinator` with a per-worktree map keyed by the
  same selector, and give `orchestration.runStop` a `worktree` param so a stop only halts the
  coordinator for that scope.
