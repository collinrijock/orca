# Issue #6698 — Vietnamese Telex input in integrated terminal loses characters (e.g. "chính" → "ch")

**Status:** NOT REPRODUCED (fixed after the report)

## Summary
The issue (filed against **v1.4.104**) reports that typing Vietnamese Telex in
the integrated terminal drops characters: typing `chinhs` (Telex for "chính")
renders only `ch`. Investigation of the current tree shows the root cause was
addressed by follow-up PRs that added a macOS **native text forwarder** for
Vietnamese input sources. The code paths that caused the loss are present and
covered; a unit test driving the real modules with the exact issue word
(`chính`) shows every character is now forwarded intact.

## Root cause (historical) and the fix now in the tree
Original cause: xterm's Kitty keyboard protocol encoder could encode and
**cancel** a printable keydown (`preventDefault`) before Chromium committed the
IME-transformed native glyph, so the committed Vietnamese text never reached the
PTY — only the earliest un-transformed keystrokes survived.

Fix now present:
- `src/renderer/src/components/terminal-pane/terminal-ime-input-source.ts:26` —
  a Vietnamese input source (`telex`/`unikey`/`vietnam`/`vni` in the source id)
  enables `forwardShortTextReplacements: true`. The macOS built-in
  `com.apple.inputmethod.VietnameseIM.VietnameseTelex` id matches.
- `src/renderer/src/components/terminal-pane/terminal-ime-native-text-candidates.ts:149`
  — with that feature on, printable ASCII letter/digit keydowns (the Telex keys)
  are treated as native-text candidates when not composing.
- `src/renderer/src/components/terminal-pane/terminal-ime-native-text-forwarder.ts:95`
  — the forwarder claims the candidate keydown (caller returns `false` from
  `attachCustomKeyEventHandler`, bypassing the Kitty encoder) and then forwards
  the committed glyph from the following `input` (`insertText`) event straight to
  the PTY (`terminal-ime-native-text-forwarder.ts:137`).
- Wired in `src/renderer/src/components/terminal-pane/use-terminal-pane-lifecycle.ts:920`.

Landed by: #6682 (Support Vietnamese IME and synthetic Unicode in terminal
forwarder), #6699 (Fix macOS IME input mode detection), #7102 (Fix ime input
switch) — all merged **after** the v1.4.104 report.

## Automated evidence
Unit test exercising the real modules (`getMacNativeTextInputSourceFeatures`,
`installTerminalImeNativeTextForwarder`) with the issue's word "chính":

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/components/terminal-pane/repro-6698-vietnamese-telex-forwarding.test.ts
```

Result: 4 passed. The "FIX CHECK" assertions confirm each glyph of `chính`
(c, h, í, n, h) is forwarded to the PTY and never truncated to `ch`; a
contrasting case shows a non-Vietnamese source does not claim the letter keydown
(the pre-fix path).

## Manual repro (for a maintainer with a live macOS app)
1. Open Orca, open a local terminal.
2. Select macOS built-in "Vietnamese – Telex" input source.
3. Type `chinhs`. Expected (current tree): `chính` renders fully.
   The bug would show only `ch`.

## Fix direction
No change required — already fixed. If a residual report appears, the two most
likely gaps are: (a) a third-party Vietnamese IME whose input-source id contains
none of `telex`/`unikey`/`vietnam`/`vni`, so `forwardShortTextReplacements` stays
off; or (b) a tone-application commit delivered as `insertReplacementText` rather
than `insertText` (the forwarder only forwards `insertText`,
`terminal-ime-native-text-forwarder.ts:144`). Either would be a new, narrower
issue.
