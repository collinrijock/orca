# Issue #7740 — Either not following CLAUDE_CONFIG_DIR or not loading shell env properly?

**Status:** REPRODUCED (root cause proven via unit test against real product code; full
new-terminal path additionally requires the live app)

## Summary

A user who exports `CLAUDE_CONFIG_DIR` (e.g. `~/.config/claude`) from their interactive
`~/.zshrc` finds that opening a Claude tab in an already-running Orca still creates and
uses the default `~/.claude`. Orca's Claude managed-account path resolver derives the
config dir **only** from `process.env.CLAUDE_CONFIG_DIR` — the login-shell snapshot the
GUI process captured at app launch. That snapshot never re-sources the user's interactive
shell per terminal/agent spawn, so a `CLAUDE_CONFIG_DIR` that lives only in `~/.zshrc`
(or was added after launch) is invisible until a full `Cmd+Q` relaunch bakes it into the
snapshot.

## Root cause

`src/main/claude-accounts/runtime-paths.ts:14-24`

```ts
getRuntimePaths(): ClaudeRuntimePaths {
  const inheritedConfigDir = process.env.CLAUDE_CONFIG_DIR?.trim() || null   // launch snapshot only
  const configDir = inheritedConfigDir || join(homedir(), '.claude')         // silent default
  mkdirSync(configDir, { recursive: true })                                  // materializes ~/.claude
  return {
    configDir,
    ...
    envPatch: inheritedConfigDir ? { CLAUDE_CONFIG_DIR: configDir } : {}      // empty when unset
  }
}
```

When `process.env.CLAUDE_CONFIG_DIR` is absent (the normal case for a GUI-launched app
whose env was snapshotted before/without the interactive `~/.zshrc` export), the resolver:

1. falls back to the default `~/.claude`,
2. `mkdirSync`-creates it — exactly the reported "it tried to create the default claude
   config directory" symptom, and
3. returns an **empty** `envPatch`, so nothing downstream re-points Claude at the user's dir.

This `envPatch` flows into the PTY spawn env for managed Claude accounts:
`runtime-auth-service.ts:698` (`envPatch: paths.envPatch`) →
`ipc/pty.ts:3076` / `ipc/pty.ts:3878` (`{ ...sshScopedEnv, ...claudeAuth.envPatch }`).

There is no per-spawn re-source of the user's shell environment anywhere in this path; the
resolver's sole source of truth is the app-launch `process.env`.

## Automated evidence

```
pnpm exec vitest run --config config/vitest.config.ts src/main/claude-accounts/repro-7740-config-dir-launch-snapshot.test.ts
```

`src/main/claude-accounts/repro-7740-config-dir-launch-snapshot.test.ts` imports the real
`ClaudeRuntimePathResolver` and:

- **BUG assertion** — with `CLAUDE_CONFIG_DIR` unset (shell-only export, not in the
  snapshot), `getRuntimePaths()` returns `configDir = <HOME>/.claude`, creates that dir on
  disk, and returns `envPatch === {}`. HOME is redirected to a temp dir so the test never
  touches the developer's real `~/.claude`.
- **CONTROL assertion** — only when `CLAUDE_CONFIG_DIR` is present in `process.env` (i.e.
  after the reporter's step-4 full relaunch) does the resolver honor it and emit
  `envPatch === { CLAUDE_CONFIG_DIR: <dir> }`.

Both pass on the current tree (2 passed).

## Manual repro (live app, macOS)

1. With Orca running, add `export CLAUDE_CONFIG_DIR="$HOME/.config/claude"` to `~/.zshrc`.
2. Open a brand-new Claude tab in the already-running instance → Claude creates/uses the
   default `~/.claude`; `echo $CLAUDE_CONFIG_DIR` may be empty at the pane too.
3. Fully `Cmd+Q` and relaunch → new Claude tabs now honor the var.

## Fix direction

- Resolve `CLAUDE_CONFIG_DIR` from a freshly-sourced login-shell env per spawn (the same
  mechanism `hydrate-shell-path.ts` uses for `PATH`, extended to carry select vars), or
- read the value the user's terminal actually resolves (the wrappers already source
  `~/.zshrc`), and seed the managed-account scoped keychain/service for that dir rather than
  materializing the unscoped default `~/.claude`.
- At minimum, avoid silently creating the default `~/.claude` and losing the user's intent
  when the var is set only in the interactive shell.

Related: cause 2 in the issue thread (#6921 / draft PR #6963) covers the managed-account
scoped-keychain seed; this reproduction isolates cause 1 (env-snapshot freshness).
