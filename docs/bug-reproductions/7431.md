# Issue #7431 — `worktree create` hard-errors offline when base ref is a stale/never-fetched remote-tracking ref

**Status:** PARTIAL (root cause proven at the unit level against real product code + real git; the full `worktreeCreate` throw is not driven end-to-end)

## Summary

When `worktree create` (no explicit `--base-branch`) resolves its base to a remote
display name such as `origin/main`, and that remote-tracking ref
(`refs/remotes/origin/main`) is **not present locally** (fresh/shallow clone, or a
default branch never fetched into this clone), the create path tries to refresh it
with a `git fetch`. Offline, that fetch fails and the create hard-errors with:

    Could not refresh base ref "origin/main" from "origin". Check your network and try again.

The bug is that a same-named **local** branch `refs/heads/main` — a perfectly usable
offline base — is never consulted before throwing. The offline fallback that the code
*intends* to provide (reuse a local base ref) is skipped because the local-base-ref
detection does not strip the `origin/` remote prefix.

## Root cause

Two pieces combine:

1. `src/main/runtime/orca-runtime.ts` `worktreeCreate` (~lines 16237-16268): the throw
   fires when `!hadRemoteTrackingBaseRef && !hasLocalBaseRef` and the refresh fetch
   fails. `hasLocalBaseRef` comes from the module-private
   `hasLocalWorktreeBaseRef(repo.path, baseBranch)` where `baseBranch === "origin/main"`.

2. `src/shared/worktree-base-ref.ts` `resolveWorktreeAddBaseRef` (lines 14-24): for a
   base containing `/`, the candidate list is
   `["refs/remotes/origin/main", "refs/heads/origin/main"]`. It probes a **literal**
   branch named `origin/main`, never `refs/heads/main`. When neither candidate exists it
   returns the unqualified `"origin/main"`, and `hasLocalWorktreeBaseRef` then falls to
   `hasCommitObjectViaGitExec("origin/main")` (`src/main/git/commit-object-ref.ts`), which
   short-circuits to `false` because `origin/main` is not a 40-char object id.

Net: `hasLocalBaseRef === false` even though `refs/heads/main` exists locally, so the
create path takes the fetch-and-throw branch instead of the "reuse local base" branch.

## Automated evidence

Unit test (real product modules + a real temp git repo with only a local `main`
branch, no `refs/remotes/origin/main`):

```
pnpm exec vitest run --config config/vitest.config.ts src/main/runtime/repro-7431-offline-remote-base.test.ts
```

Result: `Test Files 1 passed`, `Tests 5 passed`. Key pinned assertions:

- `resolveWorktreeAddBaseRef("origin/main", refExists)` probes only
  `["refs/remotes/origin/main", "refs/heads/origin/main"]` and returns `"origin/main"`
  (never tries `refs/heads/main`).
- Reconstructing `hasLocalWorktreeBaseRef`'s exact composition with the real functions
  yields `hasLocalBaseRef === false` while `hasWorktreeBaseCommitRef(repo, "refs/heads/main")`
  is `true` — the usable local base is present but ignored.
- Control: bare `"main"` resolves to `refs/heads/main` (why `--base-branch main` works),
  and an already-fetched `refs/remotes/origin/main` resolves fine (the common offline
  case that does NOT hit the bug).

The existing suite already proves the throw itself fires in this branch:
`src/main/runtime/orca-runtime.test.ts` "does not create a runtime local worktree when
the refresh fails and no local base ref exists" (~line 3535). This repro adds the
missing half: with a local `main` present, detection still reports no local base ref.

## Manual repro (from the issue)

1. Use a clone that lacks `refs/remotes/origin/main` locally (fresh shallow clone or
   default branch never fetched), but which has a local `main` branch.
2. Be offline / `origin` unreachable.
3. `orca worktree create --name demo` (no `--base-branch`).
4. Fails with `Could not refresh base ref "origin/main" from "origin".`
   Workaround: `orca worktree create --name demo --base-branch main` (a local ref).

## Fix direction

When the resolved base is `origin/<branch>` and the remote-tracking ref is absent, add
`refs/heads/<branch>` (remote prefix stripped) as a fallback candidate before declaring
no local base ref — either inside `resolveWorktreeAddBaseRef` (when the first segment is
a known remote) or in the create path's `hasLocalWorktreeBaseRef` call. Alternatively /
additionally, expose an `--offline` / `--no-fetch` flag as the issue requests. Must keep
the existing "throw when there is truly no usable local base" behavior intact.
