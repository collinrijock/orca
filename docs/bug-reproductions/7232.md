# Issue #7232 — Appearance language is changed after each install

**Status:** NOT REPRODUCED (already fixed in current tree)

## Summary
The reporter (Orca v1.4.120, Spanish OS, English UI preference) saw the UI language
revert to the system language after every update: the Appearance > Language dropdown
still showed the correct persisted choice, but the translations were not applied.

This was a real bug in v1.4.120, but it was fixed by **PR #7193**
("fix(i18n): apply persisted UI language at startup instead of OS locale"), which
merged on 2026-07-03 and shipped in **v1.4.121** — the release immediately after the
reporter's v1.4.120. The current tree contains the fix, and the regression tests that
pin the corrected behavior pass.

## Root cause (of the original bug, now fixed)
`src/renderer/src/i18n/I18nProvider.tsx` (pre-fix version):

```ts
const uiLanguage = useAppStore((state) => state.settings?.uiLanguage ?? UI_LANGUAGE_SYSTEM)
const locale = resolveUiLocale(uiLanguage)
useEffect(() => {
  if (i18n.language !== locale) {
    void i18n.changeLanguage(locale)
  }
}, [locale])
```

Two defects combined:
1. While settings loaded async over IPC, `state.settings` is `null`, so `uiLanguage`
   fell back to `'system'`. On a Spanish OS that resolved to `es` and kicked off a
   `changeLanguage('es')` for the OS locale.
2. The `i18n.language !== locale` guard compared against a value that stays **stale**
   during the in-flight lazy catalog load, so when the persisted English preference
   arrived it was skipped and the in-flight OS-locale switch won. Result: correct
   dropdown value, wrong applied language — exactly the report.

## Fix in current tree
`src/renderer/src/i18n/I18nProvider.tsx:13-26` now:
- reads `state.settings?.uiLanguage ?? null` and applies **nothing** until settings
  actually load (no OS-locale race), and
- tracks the last *requested* locale in a ref instead of reading the stale
  `i18n.language`, so it always corrects back to the persisted preference.

Persistence side is also sound: `src/main/persistence.ts:3206`
(`uiLanguage: normalizeUiLanguage(parsed.settings?.uiLanguage)`) preserves the stored
value across updates, and `src/main/index.ts:2129`
(`setMainUiLanguage(store.getSettings().uiLanguage)`) applies it to the main process at
startup.

## Automated evidence
The existing regression suite pins the fixed startup ordering, including the exact
non-English-OS scenario from this report (`#7188`):

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/i18n/I18nProvider.test.tsx
```

Result on current tree: **5 passed** — notably
"never applies the OS locale on a non-English system when English is persisted (#7188)"
which asserts `changeLanguage` is called with `'en'` and never `'es'`.

## Manual repro (would apply to v1.4.120 only)
1. On a non-English OS, set Orca UI language to English.
2. Update Orca and relaunch.
3. Pre-fix: UI shows system (Spanish) language while dropdown shows English.
   Current tree: UI stays English.

## Fix direction
None required — fixed by #7193. Recommend closing #7232 as resolved/duplicate of the
#7188/#7193 fix, ideally confirming with the reporter on >= v1.4.121.
