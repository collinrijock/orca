# Issue #6905 — Vietnamese IME Input Broken in Terminal

**Status:** NOT REPRODUCED (current tree already handles it; full runtime confirmation needs LIVE_APP_MAC)

## Summary
The issue reports that typing Vietnamese (Telex/VNI, e.g. "xin chào", "tiếng việt")
in Orca's integrated terminal produces garbled, duplicated, or dropped characters.
On the **current tree** the terminal input path has a dedicated, well-tested macOS
IME text-forwarding subsystem that handles exactly these Vietnamese scenarios. The
reported bug does not reproduce at the logic level.

## What I verified
- Vietnamese IME support landed in `Support Vietnamese IME and synthetic Unicode in
  terminal forwarder` (#6682, 2026-06-28), with follow-ups `Fix macOS IME input mode
  detection` (#6699) and `Fix ime input switch` (#7102, 2026-07-05). The issue was
  filed 2026-06-30, i.e. it predates the follow-up fix and reads as a generic/templated
  report (it lists Linux iBus Bamboo and Windows Unikey even though the OS field is macOS).
- The current input path:
  - `getMacNativeTextInputSourceFeatures()` recognizes Telex/VNI/Unikey/"vietnam" input
    sources and enables `forwardShortTextReplacements`, so transformed letters/digits are
    forwarded from the `input` event rather than translated by xterm's kitty encoder.
  - `installTerminalImeNativeTextForwarder()` claims the printable keydown (so xterm stays
    silent), bypasses the matching keypress/keyup (so no double-send and no kitty release
    leak), and forwards the committed glyph exactly once from the `insertText` input event.
  - Composition-driven input is explicitly excluded from the forwarder (left to xterm's
    `CompositionHelper`), and `terminal-ime-xterm-composition-deduplication.test.ts` pins
    the de-duplication behavior so a committed glyph is not emitted twice.
- The existing suites pass: `terminal-ime-native-text-forwarder.test.ts`,
  `terminal-ime-input-source.test.ts`,
  `terminal-ime-xterm-composition-deduplication.test.ts` (54 tests).
- My repro harness drives the **real** modules through the issue's own scenarios
  ("xin chào", VNI "đ", composition guard) and every glyph reaches the PTY exactly once,
  in order, with no duplication or dropped diacritics.

## Root cause
No defect found in current code. The reported behavior corresponds to the pre-#6682 state
where the terminal had no Vietnamese-aware native-text forwarding, so the kitty keyboard
encoder cancelled the printable keydown before Chromium committed the transformed glyph.
Relevant current code:
- `src/renderer/src/components/terminal-pane/terminal-ime-input-source.ts:26` — Vietnamese
  feature gate (`forwardShortTextReplacements`) and `:57` source-term matching.
- `src/renderer/src/components/terminal-pane/terminal-ime-native-text-forwarder.ts:95` —
  keydown claim + `:137` committed-text forward from the `input` event.
- `src/renderer/src/components/terminal-pane/terminal-ime-native-text-candidates.ts:121` —
  `isImeNativeTextKeydownCandidate` decision.
- `src/renderer/src/components/terminal-pane/use-terminal-pane-lifecycle.ts:920` — wiring
  (mac-only) into `attachCustomKeyEventHandler`.

## Manual repro (needs live macOS + Vietnamese IME)
1. macOS System Settings → Keyboard → add "Vietnamese – Telex" (or VNI, or Unikey).
2. Launch Orca, focus the integrated terminal, switch the input source to Vietnamese.
3. Type "xin chào" and "tiếng việt".
4. Expected (and observed logic-level): characters compose correctly with no duplication.
   If garbling is still seen on a real host, capture the input-source id reported by
   `app:getKeyboardInputSourceId` and the DOM `input`/`composition` event sequence.

changing the forwarder mechanics.
