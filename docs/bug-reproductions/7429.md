# Issue #7429: `worker_done` with null taskId/dispatchId silently fails to complete the task

**Status:** REPRODUCED

## Summary

`reconcileWorkerDoneMessage` drops a `worker_done` message that lacks a `taskId`
or `dispatchId` in its payload — returning `{ action: 'ignored' }` with only a
warning log — **before** it ever performs the terminal/pane ownership check. So a
worker that emits `worker_done` from the exact pane that owns the sole live
dispatch, but does not embed the ids (custom harness, hand-crafted
`--payload '{}'`, or a worker that did not run the stock preamble verbatim), never
auto-completes its task. There is no error surfaced to the sender; the task just
stays in `dispatched`/`running` forever.

## Root cause

`src/main/runtime/orchestration/lifecycle-reconciliation.ts`, in
`reconcileWorkerDoneMessage`:

- Line 173-177: if `payload.taskId` is not a non-empty string → `return { action: 'ignored' }`.
- Line 179-183: if `payload.dispatchId` is not a non-empty string → `return { action: 'ignored' }`.

Both early returns run before the `hasLifecycleAuthority(dispatch, msg)` pane/
handle-ownership check at line 204. Because the ids are the sole key used to look
up the dispatch, there is no fallback to reconcile a null-id `worker_done`
against the single in-flight dispatch owned by `msg.from_handle` /
`msg.sender_pane_key`. The ids live only as opaque CLI flags assembled into the
JSON payload — they are not on the `SendParams`/`MessageRow` schema and are never
validated at `send` time, so a misbehaving worker fails silently rather than
loudly.

## Automated evidence

Unit test that imports the real `OrchestrationDb` and `reconcileLifecycleMessage`
and pins the buggy behavior (null-id `worker_done` from the owning pane is
`ignored` and the task stays `dispatched`), plus a control proving the identical
message *with* the ids completes the task:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/runtime/orchestration/repro-7429-worker-done-null-ids.test.ts
```

Result: 3 passed. The `BUG:` tests assert the current wrong result; commented-out
assertions show the correct expected behavior (`action: 'completed'`, task
`completed`).

## Manual repro (app steps)

1. `orca orchestration task-create --title demo` → `task_<id>`
2. `orca orchestration dispatch --task <id> --terminal <handle> --inject`
3. From the dispatched worker, emit `worker_done` without the matching ids (e.g.
   a custom harness that sends `--payload '{}'`).
4. `orca orchestration task-list` → the task never auto-completes; it must be
   closed by hand with `orca orchestration task-update --status completed`.

## Fix direction

Per the issue's "Expected", highest-leverage options:

- When a `worker_done` (or other lifecycle message) arrives with null/missing
  ids, fall back to reconciling against the single in-flight dispatch owned by
  `msg.from_handle` / `msg.sender_pane_key` (via `hasLifecycleAuthority`) and
  complete that dispatch — instead of returning `ignored` before the ownership
  check. Only ambiguous cases (zero or >1 in-flight dispatch for the sender)
  should be dropped.
- And/or reject null-id lifecycle messages at `send` time with a clear error so
  the worker fails loudly.

## Key files

- `src/main/runtime/orchestration/lifecycle-reconciliation.ts` (reconcileWorkerDoneMessage, lines 173-183)

- `src/main/runtime/orchestration/preamble.ts` (buildDispatchPreamble — embeds the ids)
