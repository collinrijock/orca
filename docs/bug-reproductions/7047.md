# Issue #7047 — Agent is marked "Done" when the CLI is not installed

**Status:** REPRODUCED

## Summary

On an Orca server (headless serve), starting a Codex/Claude agent when the CLI
binary is missing still surfaces an agent row to the web/mobile client and marks
it **Done**, even though the process reported `command not found` and never ran.
The status derivation that builds the per-tab `agentStatus` for paired clients
only inspects the last *title-derived* agent status; it never consults the PTY's
exit code. Any agent that is not currently `working`/`permission` collapses to
`done`, so a failed/never-started agent is reported identical to one that
completed successfully. There is no `failed` state anywhere in the derivation.

## Root cause

`OrcaRuntimeService.buildPtyMobileAgentStatus` in
`src/main/runtime/orca-runtime.ts` (the derivation that produces the
`agentStatus` every headless-server client renders):

```
src/main/runtime/orca-runtime.ts:22530
    state:
      pty!.lastAgentStatus === 'working'
        ? 'working'
        : pty!.lastAgentStatus === 'permission'
          ? 'blocked'
          : 'done',              // <-- everything else, including a failed launch, becomes 'done'
```

The PTY record already carries `lastExitCode` (`RuntimePtyWorktreeRecord`,
`src/main/runtime/orca-runtime.ts:1027`) — it is `127` for `command not found` —
but the derivation never reads it. The same "default to done" collapse also
exists in the renderer-synced branch of `toMobileSessionTabsResult`
(`src/main/runtime/orca-runtime.ts:22368-22383`), which resets a launched
agent's status to `state: 'done'` once the shell reclaims the pane title.

Grep confirms no code path ever *produces* an `AgentStatusState` / `AgentDotState`
of `'failed'` — it exists only as a display label
(`src/renderer/src/components/sidebar/worktree-card-agent-summary.ts:44`).

## Automated evidence

Unit test imports the real `OrcaRuntimeService` and calls the real private
derivation `buildPtyMobileAgentStatus` with a PTY record shaped exactly like a
launched-but-missing Codex agent (`launchAgent: 'codex'`, `lastExitCode: 127`,
`lastAgentStatus: 'idle'`). It asserts the client is told the agent is `done`,
and that flipping the exit code between `127` (failed) and `0` (clean) makes no
difference — proving the derivation is exit-code-blind.

Re-run command:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/runtime/repro-7047-cli-missing-marked-done.test.ts
```

Result: 2 tests pass, pinning the buggy `done` derivation on the current tree.

## Manual repro (from the issue)

1. Set up a fresh Orca server, do **not** install `codex` (or `claude`).
2. From a paired web/mobile client, start a Codex/Claude agent.
3. Open the terminal — it shows `zsh: command not found: codex`.
4. The sidebar still shows a Codex agent entry, marked **Done**.

## Fix direction

Consult the PTY exit signal when deriving the terminal agent status. When a
launched agent's PTY exits non-zero (notably `127` = command not found) without
ever reaching a real `working`/hook state, derive a `failed` state (add
`'failed'` to `AgentStatusState`) instead of defaulting to `done`, or suppress
the agent row entirely so a never-started agent is not presented as a completed
one. Apply the same treatment to the `toMobileSessionTabsResult` reset branch and
mirror it in the renderer's agent-row derivation so desktop and server agree.

## Key files

- `src/main/runtime/orca-runtime.ts:22530` — `buildPtyMobileAgentStatus` default-to-`done` derivation (primary root cause)
- `src/main/runtime/orca-runtime.ts:22368-22383` — synced-tab branch resets a launched agent to `state: 'done'`
- `src/main/runtime/orca-runtime.ts:1027` — `RuntimePtyWorktreeRecord.lastExitCode` (available but ignored)
- `src/renderer/src/components/sidebar/worktree-card-agent-summary.ts:44` — `'failed'` exists only as a label, never produced
- `src/main/runtime/repro-7047-cli-missing-marked-done.test.ts` — the reproduction
