# Issue #7038 — Claude Code terminal auto-selects "1" on refocus

**Status:** NOT REPRODUCED (no focus-time Enter injection in the current tree)

## Summary

Reporter (via Discord): "Sometimes when I click to go back to a Claude Code
terminal waiting for user input it seems to automatically select 1. It's
intermittent and I haven't noticed any particular set of steps that will trigger
it."

Triage hypothesis: refocus injects a stray Enter / synthetic keystroke that
submits the default option (1).

I read the real renderer focus/reattach/replay code and could not find any path
that turns regaining focus into an Enter (`\r`). At focus time the code deals
only with terminal **focus reports** (`\x1b[I` / `\x1b[O`) — never a carriage
return — and the code is specifically hardened against emulator replies leaking
into an agent's stdin. A runnable unit test exercising the real focus modules
confirms refocus injects no bytes.

## What I verified (root cause analysis)

The only ways focus influences bytes sent to the shell:

1. **xterm's native focus reporting.** When an app enables DECSET 1004
   (`\x1b[?1004h`), xterm emits `\x1b[I` on focus-in and `\x1b[O` on focus-out
   through `onData`. Orca forwards these to the PTY. This is standard terminal
   behavior and is a *focus event*, not Enter.
   - `src/renderer/src/components/terminal-pane/pty-connection.ts:307-309`
     defines the focus-in/out sequences.
   - `src/shared/terminal-query-reply.ts:56` — focus reports are **not**
     classified as query replies, so they travel the ordinary, order-preserving
     input path byte-for-byte; nothing rewrites them into a submit.

2. **Reattach focus-in re-send.** `pty-connection.ts:4975-4979` may send
   `\x1b[I` once during a reattach replay, and only when
   `terminalHasFocusReportingEnabled` **and** the pane owns DOM focus **and** a
   live-agent signal is present (`shouldSendFocusedAgentReattachFocusIn`,
   line 1727). Again this is a focus report, gated tightly — not Enter.

3. **Replay auto-reply guard.** When Orca replays recorded PTY bytes on
   reattach/restore/visibility-resume, xterm auto-answers embedded query
   sequences (CPR/DA/OSC/etc.) via `onData`. `replay-guard.ts` drops `onData`
   while a per-pane replay counter is non-zero precisely so those replies cannot
   land on the shell/agent stdin as stray input — the module comment explicitly
   calls out that "a leaked ESC reads as the user pressing Escape"
   (`replay-guard.ts:47-49`). The guard release is probe-certified (FIFO), never
   time-based, so it does not open mid-parse.

4. **Focus reclaim modules.** `regular-terminal-focus-ownership.ts` and
   `terminal-ime-input-context-refresh.ts` (the code that reclaims focus after
   window blur / rebuilds the macOS NSTextInputContext) accept **no PTY input
   sink** and only call `.focus()` / `.blur()` / a focus-state mirror. They
   structurally cannot inject a keystroke.

No path converts a refocus (or a focus report) into `\r`. The intermittent
"select 1" the reporter saw is most plausibly Claude Code's own handling of the
focus-in report it opted into, or a mouse click under app-enabled mouse
reporting — neither of which is an Orca logic defect reproducible in this tree.

## Automated evidence

A unit test drives the real focus modules and the real byte classifier and pins
the current (safe) behavior — refocus injects no bytes, focus reports are never
Enter:

```
pnpm exec vitest run --config config/vitest.config.ts src/renderer/src/components/terminal-pane/repro-7038-focus-selects-one.test.ts
```

Result: 4 passed. Key assertions:
- `\x1b[I` / `\x1b[O` are not `\r` and contain no `\r`.
- `isTerminalQueryReply('\x1b[I')` / `('\x1b[O')` are `false` (forwarded as-is).
- `resyncTerminalFocusForWindowFocus(...)` reclaims focus with only a DOM
  `.focus()` + focus-state mirror; no pty write occurs.
- `refreshTerminalImeInputContext(...)` (macOS blur+refocus) emits no pty byte.

## Manual repro (not reproduced)

Attempted mentally against the code: open a Claude Code prompt showing a 1/2/3
selection, click away, click back. The refocus emits at most `\x1b[I` (only if
Claude armed DECSET 1004). No Orca code sends `\r` on that transition.

## Fix direction (if it resurfaces)

If a real reproduction is captured live (macOS), instrument the PTY input stream
at focus time to record the exact bytes sent, to distinguish:
- Orca forwarding a focus report `\x1b[I` that Claude then mishandles (upstream
  Claude Code issue), vs.
- a mouse-button report from the focusing click under app mouse mode (arguably
  expected terminal behavior; could be suppressed for focus-only clicks), vs.
- an actual `\r` (which would be a genuine Orca defect — none found here).
