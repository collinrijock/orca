# Issue #5370 ‚Äî Orca's private `CODEX_HOME` causes a Codex auth-revocation war with `~/.codex`

**Status:** PARTIAL (design fact reproduced in code; full server-side revocation loop needs a live OpenAI account)

## Summary

Orca launches Codex under a **private, Orca-owned `CODEX_HOME`**
(`<userData>/codex-runtime-home/home`) instead of the user's canonical
`~/.codex`. That private home carries its **own `auth.json`** ‚Äî a second,
independent Codex/ChatGPT OAuth session on the same OpenAI account. Because
ChatGPT OAuth uses single-use refresh tokens (one active session per account),
the private home and `~/.codex` mutually revoke each other's refresh tokens as
either side refreshes, breaking Codex auth for everything outside Orca (launchd
jobs, plain `codex exec`, other tooling) while `codex login status` still says
"Logged in."

The **precondition** for the war ‚Äî two distinct homes, each with its own
`auth.json`, both injected/used for the same account ‚Äî is fully code-verifiable
and is what this reproduction pins. The actual server-side token revocation is
OpenAI behavior and requires a live account, hence PARTIAL rather than
REPRODUCED.

## Root cause

- `src/main/codex/codex-home-paths.ts:35` ‚Äî `getOrcaManagedCodexHomePath()`
  returns `<userData>/codex-runtime-home/home`, a private directory, **not**
  `~/.codex`.
- `src/main/codex/codex-home-paths.ts:31` ‚Äî `getSystemCodexHomePath()` returns
  `~/.codex` (the canonical home used by every non-Orca Codex invocation).
- This private home is injected as `CODEX_HOME` into every Orca-launched Codex
  process and re-asserted after shell rc files run:
  `src/main/providers/local-pty-shell-ready.ts:154-155` (`export CODEX_HOME="${ORCA_CODEX_HOME}"`),
  wired via `prepareCodexRuntimeHomeForLaunch` ‚Üí `CodexRuntimeHomeService.prepareForCodexLaunch`
  (`src/main/index.ts:757`, `src/main/index.ts:2082`).
- The private home keeps its **own** `auth.json`
  (`CodexRuntimeHomeService.getRuntimeAuthPath()`,
  `src/main/codex-accounts/runtime-home-service.ts:1091`), separate from
  `~/.codex/auth.json`.

The current tree adds substantial mitigation ‚Äî it mirrors `~/.codex/auth.json`
into the runtime home and reads refreshed tokens back
(`syncRuntimeAuthWithSystemDefault`, `readBackRefreshedTokens` in
`runtime-home-service.ts`). But that sync is **copy-based and only runs at
discrete trigger points** (launch, rate-limit fetch, account switch). Between
syncs ‚Äî and for the background daemon session the report calls out ‚Äî the two
homes still hold divergent single-use refresh tokens, which is exactly the
revocation window. The fundamental design (two homes, two `auth.json` copies)
that the report flags is unchanged; the issue's preferred fixes (#1 point
runtime at `~/.codex`; #2 share/symlink one `auth.json`) are not implemented.

## Automated evidence

A unit test imports the **real** product path resolvers and pins the buggy
design: the injected `CODEX_HOME` is a separate tree from `~/.codex`, and each
home holds its own `auth.json` with a different refresh token for the same
account.

Re-run:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/codex/repro-5370-private-codex-home.test.ts
```

Result: 2 passed. The assertions marked `BUG:` encode the wrong (current)
behavior; a fix per the issue would make `managedHome === systemHome` (or one
shared `auth.json`) and flip them.

## Manual repro (from the issue)

1. `codex login` so `~/.codex/auth.json` is valid; confirm `codex exec "say ok"`.
2. Open Orca; let it start its managed Codex runtime (private `CODEX_HOME`).
3. From a non-Orca context run Codex over time
   (`env -u CODEX_HOME codex exec "say ok"`).
4. Re-auth in either home or let both refresh. Observe `~/.codex` tokens get
   `refresh token was revoked` / `refresh token already used` while
   `codex login status` keeps reporting "Logged in."

## Fix direction

Per the issue, in order of preference:
1. Default Orca's Codex runtime to the user's `~/.codex` (share everything, not
   just `skills`/`plugins`).
2. If isolation is intentional, share `auth.json` with the canonical home
   (symlink or single source of truth) so there is one session to revoke.
3. Add a setting / CLI flag to point Orca's Codex runtime at a chosen
   `CODEX_HOME`.
4. At minimum, detect the conflict, warn, and document that Orca maintains a
   separate Codex session.
