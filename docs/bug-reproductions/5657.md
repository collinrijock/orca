# Issue #5657 — macOS startup PATH probe hangs under Endpoint Security agents

**Status:** PARTIAL (root cause proven in code + unit test; the actual kernel wedge needs a managed Mac)

## Summary

On macOS, Orca discovers the interactive `PATH` at startup by spawning the user's
login shell as an **interactive** login shell (`zsh -ilc ...`) through
`hydrateShellPath`. An interactive shell sources the user's full `~/.zshrc`, which
commonly forks many subprocesses (oh-my-zsh, `compinit`, `source <(tool completion
zsh)`, nvm, sdkman, ...). On a managed Mac, every nested `exec`/`posix_spawn` must
be authorized by each active Endpoint Security extension (Jamf Protect,
CrowdStrike Falcon) via `ES_EVENT_TYPE_AUTH_EXEC`. Under those agents the verdicts
stall and the spawn wedges in the kernel uninterruptibly (process state `U`),
which no `kill -9` / Force Quit can reap — the only recovery is a reboot.

## Root cause

`src/main/startup/hydrate-shell-path.ts:91`

```ts
const child = spawn(shell, ['-ilc', command], { ... })
```

The `-ilc` flag bundle requests an **interactive** (`i`) login (`l`) command (`c`)
shell. The `i` is the trigger: it forces evaluation of the spawn-heavy interactive
rc-file init that stalls under AUTH_EXEC. The issue's suggested fix #2 is to probe
without a full interactive shell — a non-interactive login probe (`-lc`) still
sources login profiles/`.zprofile` but skips `compinit`/completion/plugin
subprocess spawns entirely (the reporter measured the probe dropping from ~1.7s to
~0.08s, spawning nothing).

Note the file already has a 5s `SPAWN_TIMEOUT_MS` + SIGKILL fallback
(`hydrate-shell-path.ts:119`), and `index.ts:463` fires the probe non-blocking
(`void hydrateShellPath().then(...)`). Neither helps the reported failure: a
process wedged in kernel `posix_spawn` (state `U`) ignores SIGKILL, and the
maintainer already noted the mocked-timeout test "does not reproduce a
kernel-blocked posix_spawn/uninterruptible state." So the residual, real defect in
the current tree is the **interactive** probe strategy itself, which this repro
pins.

## Automated evidence

A unit test drives the REAL `spawnShellAndReadPath` path (no spawner override) and
observes the exact argv the shipped code passes to `child_process.spawn`. It PASSES
on the current tree while asserting the buggy `-ilc` interactive flags; the marked
assertions flip once the probe is made non-interactive.

Re-run:

```
pnpm exec vitest run --config config/vitest.config.ts src/main/startup/repro-5657-interactive-probe.test.ts
```

Result: `Test Files 1 passed (1)`, `Tests 2 passed (2)` — confirming the probe is
still invoked with `-ilc` (interactive).

## Manual repro (needs affected environment)

1. macOS machine with an active ES agent (verify: `systemextensionsctl list` shows
   `com.jamf.protect.security-extension` / `com.crowdstrike.falcon.Agent`).
2. A `~/.zshrc` that spawns subprocesses at interactive startup (oh-my-zsh,
   `compinit`, `source <(tool completion zsh)`, nvm, sdkman).
3. Launch Orca from Dock/Finder → hangs with no window; `ps -o pid,state,command`
   shows the Orca process and a `/bin/zsh -ilc ... __ORCA_SHELL_PATH__ ...` child
   in state `U`; unkillable, requires reboot.

## Fix direction

- Change the probe from `-ilc` to a **non-interactive** login shell `-lc` in
  `hydrate-shell-path.ts:91` so login profiles are still sourced but no
  compinit/completion/plugin subprocesses are spawned.
- Optionally add a defense-in-depth env guard the docs can point users to
  (`ZSH_EXECUTION_STRING` discriminator) — but that is a user-side workaround, not
  the Orca fix.
- The existing timeout + SIGKILL remains useful for slow (not wedged) rc files but
  cannot recover a kernel-`U` process, so it is not a substitute for dropping `-i`.
